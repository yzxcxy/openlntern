# OpenViking 后端重构方案

## 背景

当前后端把 OpenViking 的接入逻辑同时放在了 `services`、`controllers` 和部分 middleware 中，导致：

- 客户端实例初始化与 HTTP 细节泄漏到业务层。
- 控制器直接拼装 OpenViking URI、解析返回结构，职责混杂。
- 上层调用方法名直接暴露 `openviking`，业务语义和底层存储实现耦合。
- 同一套条目解析、路径拼接和错误判断逻辑散落重复。

## 重构目标

- 将 OpenViking 视为一种“远程上下文数据库”，实例统一收敛到 `internal/database`。
- 将资源访问、结果解析、URI 组装等数据访问逻辑下沉到 `internal/dao`。
- 控制器、Agent middleware 等上层仅调用知识库/技能等业务语义接口。
- 删除重复的 OpenViking 工具函数和已失效的本地技能扫描冗余代码。

## 目标分层

### 1. database 层

- 新增 `internal/database/context_store.go`
- 统一持有远程上下文库实例：`database.Context`
- 负责：
  - 读取 `tools.openviking` 配置
  - 初始化 HTTP Client
  - 封装 `GET / POST / DELETE` 请求
  - 处理通用日志与鉴权头

这一层只负责“拿到数据库连接并发请求”，不承载知识库/技能业务语义。

### 2. dao 层

- 新增 `internal/dao/context_store.go`
  - 统一解析 OpenViking 响应结构
  - 统一把远程条目转换为 `ResourceEntry`
- 新增 `internal/dao/knowledge_base.go`
  - 对外提供 `KnowledgeBase` DAO
  - 以“知识库”语义暴露能力：列出、查看树、导入、移动、删除
- 新增 `internal/dao/skill_store.go`
  - 对外提供 `SkillStore` DAO
  - 以“技能仓库”语义暴露能力：列技能、列文件、读文件、读摘要、导入、删除

这一层屏蔽 OpenViking 的接口路径、响应格式、URI 拼接细节。

### 3. controller / service / middleware 层

- 控制器改为调用 `dao.KnowledgeBase` 与 `dao.SkillStore`
- Agent middleware 改为依赖 `SkillRepository` 业务接口，而不是 OpenViking 客户端细节
- 上层不再直接调用 `OpenVikingList / OpenVikingReadContent / OpenVikingMove` 等名字

## 本次已落地的重构点

### database

- 将 OpenViking 客户端实例从 `services` 迁移到 `database.Context`
- 启动入口改为在 `main.go` 中调用 `database.InitContextStore`

### dao

- 统一下沉：
  - 目录列举
  - 树查询
  - 内容读取
  - 摘要读取
  - 资源导入
  - 路径移动
  - 删除
- 为知识库与技能分别提供面向业务的 DAO

### 上层调用

- `controllers/kb.go` 改为调用 `dao.KnowledgeBase`
- `controllers/skill.go` 改为调用 `dao.SkillStore`
- `services/agent.go` 与 `services/middlewares/skill` 改为依赖通用技能仓库接口

### 冗余删除

- 删除原 `internal/services/openviking.go`
- 删除 `controllers/skill.go` 中不再参与主流程的本地技能目录扫描辅助函数

## 设计原则

- 上层只表达业务：知识库、技能、技能文件。
- 中层负责数据访问：DAO 屏蔽底层 OpenViking 细节。
- 底层只负责连接：database 层只持有客户端和基础请求能力。
- 返回结构尽量收敛为稳定的 typed struct，减少 `map[string]any` 向上扩散。

## 后续建议

- 为 `SkillFrontmatter` 再补一个 DAO，彻底统一“数据访问都走 DAO”。
- 把 `isStoreNotFound` 这类错误识别收敛成 DAO 内的 typed error。
- 为知识库和技能 DAO 增加单元测试，避免继续依赖真实外部环境做集成校验。
- 如果后续引入其他远程上下文存储，可直接复用当前的 database/dao 分层模式。
