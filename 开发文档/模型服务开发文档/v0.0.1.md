# openIntern 模型服务开发方案

查阅日期：2026-02-28

本文档只整理调研结果和具体改造方案，当前阶段不直接落实现。

## 1. 目标

为 openIntern 增加“模型服务”能力，覆盖两部分：

1. 前端增加模型服务管理入口，并在聊天页支持模式切换与模型选择。
2. 后端通过 AG-UI 协议的 `forwardedProps` 接收模型路由信息，从已存储的模型配置中解析出实际可用模型，再按 provider 选择对应的 Eino `ChatModel` 实现。

本期范围内：

1. `agent` 模式仅做占位，不接实际执行链路。
2. 模型密钥不从前端透传，前端只传“选择结果”，后端再读取数据库中的模型配置。

## 2. 外部调研结论

### 2.1 Semi Design 可满足聊天输入区扩展

当前项目已经在 `openIntern_forentend/app/(workspace)/chat/page.semi.tsx` 中使用 `@douyinfe/semi-ui-19` 的 `AIChatInput`。

查阅 Semi 官方文档后，`AIChatInput` 已提供两类扩展点，正好覆盖本次需求：

1. `renderConfigureArea`
   用于在输入框底部渲染“配置区”，适合放置“Chat 模式 / Agent 模式”的上拉选择。
2. `renderActionArea`
   用于在输入区右侧渲染动作区，适合在 `chat` 模式下放置“模型选择”按钮。

这意味着不需要重写输入框，只需要在现有 `AIChatInput` 上增加定制渲染。

### 2.2 AG-UI 已原生支持 `forwardedProps`

AG-UI 的 `RunAgentInput` 原生包含 `forwardedProps` 字段，用于透传额外属性。

项目内本地 Go SDK `go/pkg/core/types/types.go` 也已实现：

1. `RunAgentInput.ForwardedProps any`
2. 同时兼容 `forwardedProps` 与 `forwarded_props`

因此，本次不需要扩展 AG-UI 协议结构，只需要约定新的 `forwardedProps` 字段内容。

### 2.3 Eino 适合按 API 类型做 ChatModel 路由

查阅 CloudWeGo Eino 官方文档后，`ChatModel` 已有多种生态集成能力，包含当前项目已使用的：

1. `ark`
2. `deepseek`

同时文档也列出了多种其他可选实现（以官方集成列表为准），例如：

1. `claude`
2. `gemini`
3. `ollama`
4. 其他兼容的模型接入实现

但这里不适合“每新增一个模型提供商，就预置一套单独实现”。

更合理的做法是按 API 类型分层：

1. `ark`
   一类实现，对应火山方舟协议族。
2. `openai_compatible`
   一类实现，覆盖 OpenAI 兼容接口族。
3. `anthropic`
   一类实现，覆盖 Claude 接口族。
4. `gemini`
   一类实现，覆盖 Gemini 接口族。
5. `ollama`
   一类实现，覆盖本地/自托管 Ollama 接口族。

结论是：后端应维护少量“API 类型 -> Eino ChatModel 装配器”，模型提供商只是在数据层声明自己属于哪种 API 类型，而不是和 Eino 实现一一绑定。

### 2.4 主流 provider 方案应拆为“提供商”和“模型”两层

综合现有模型提供商，技术上大致可分为三类：

1. OpenAI 兼容接口
   典型特征是 `base_url + api_key + model`，适合统一封装，DeepSeek、部分聚合平台、部分私有代理都属于这一类。
2. 厂商原生接口
   如 Ark、Claude、Gemini，通常仍然是“凭证 + 模型名 + 可选 endpoint/region”的组合。
3. 本地/自托管接口
   如 Ollama，通常不需要外部密钥，但需要 `base_url` 与模型清单。

对 openIntern 来说，最稳妥的方案不是把“提供商配置 + 模型列表”塞进同一条记录，而是拆成三层：

1. 模型提供商
   存放 `api_type`、`api_key`、`base_url`、提供商名称、提供商头像等连接级信息。
2. 模型
   存放模型名称、模型标识、所属提供商、模型头像、能力标签等模型级信息。
3. 系统默认模型
   单独存放当前系统默认使用的模型，不挂在某个提供商配置上。

这样可以把“连接配置”和“可选模型目录”明确解耦，后续扩展更稳定。

## 3. 当前项目现状

### 3.1 前端现状

1. 快捷入口位于 `openIntern_forentend/app/(workspace)/layout.tsx`
   目前已有 `A2UI 管理`、`Skill 市场`、`知识库` 等入口按钮，新增“模型服务”入口应复用同一块布局。
2. 聊天页实际入口是 `openIntern_forentend/app/(workspace)/chat/page.tsx`
   当前只是转发到 `page.semi.tsx`。
3. 聊天输入区位于 `openIntern_forentend/app/(workspace)/chat/page.semi.tsx`
   已经渲染 `AIChatInput`，发送逻辑在 `handleMessageSend` 中通过 `agent.runAgent()` 触发。
4. 当前 `CopilotKitProvider` 指向 `/api/copilotkit`
   再由 `app/api/copilotkit/[[...slug]]/route.tsx` 中的 `HttpAgent` 转发到后端 `/v1/chat/sse`。

### 3.2 后端现状

1. SSE 入口为 `openIntern_backend/internal/controllers/chat.go`
   请求体直接绑定到 `types.RunAgentInput`。
2. 实际执行链路在 `openIntern_backend/internal/services/agent.go`
   顺序为：
   `绑定输入 -> 合并历史 -> applyForwardedPropsChain -> AGUI 转 Eino 消息 -> runEinoStreaming`
3. 当前 `applyForwardedPropsChain` 只处理 `a2uiAction`
   这说明这里已经是“解析 forwardedProps”的正式插入点。
4. 当前模型在 `InitEino` 中全局初始化
   现在是固定使用一个 `ark` 模型作为主模型，`deepseek` 作为标题总结模型，尚不支持按请求动态切换。

### 3.3 现有约束

1. 现有 `einoRunner` 是全局单例，不适合直接按请求动态切模型。
2. 当前数据库自动迁移只包含：
   `User`、`A2UI`、`Thread`、`Message`、`SkillFrontmatter`
3. 还没有模型配置管理表，也没有模型管理 API。

## 4. 推荐的整体设计

### 4.1 核心原则

1. 前端只传“选择结果”，不传密钥。
2. 提供商配置、模型目录、系统默认模型三者分开管理。
3. `forwardedProps` 使用单一总键承载智能体运行配置，内部再分级。
4. 后端按 `APIType` 选择 Eino 装配器，而不是按每个 provider 硬编码。
5. 模型实例按请求解析，不再把主对话模型固定死在全局单例里。

### 4.2 `forwardedProps` 新约定

推荐新增如下结构：

```json
{
  "agentConfig": {
    "conversation": {
      "mode": "chat"
    },
    "model": {
      "providerId": "9b08f9d2-9c8f-4b5b-b8fb-4c0b9d65d4f2",
      "modelId": "6fef6f48-4c0b-45b2-9831-cff6b708ff74"
    },
    "features": {}
  }
}
```

字段说明：

1. `agentConfig`
   作为智能体运行配置的总键，和当前的 `a2uiAction` 一样是顶层语义键。
2. `agentConfig.conversation`
   承载对话模式等会持续扩展的对话级配置。
3. `agentConfig.model`
   承载模型路由信息，建议传 `modelId`，`providerId` 作为冗余校验或快速定位字段。
4. `agentConfig.features`
   预留给后续能力开关、上下文策略、推理偏好等扩展项。

注意：

1. 不要在 `forwardedProps` 中放 `api_key`、`base_url` 等敏感信息。
2. `agent` 模式本期仅占位，前端可以先发送 `agentConfig.conversation.mode=agent`，但在 UI 层直接提示“暂未开放”，避免误触发真实对话。
3. 后续如果接入其他功能，继续在 `agentConfig` 下新增分组，而不是继续扩展顶层散键。

### 4.3 模型配置存储结构（推荐 MVP）

为了兼顾后续扩展与数据边界清晰度，建议从一开始就拆成“提供商表 + 模型表 + 默认配置表”。

建议新增模型 `ModelProvider`，字段如下：

1. `ID uint`
   内部自增主键。
2. `ProviderID string`
   对外业务 ID，UUID。
3. `Name string`
   展示名称，如“默认 DeepSeek 网关”“生产方舟”。
4. `APIType string`
   如：`ark`、`openai_compatible`、`anthropic`、`gemini`、`ollama`。
5. `BaseURL string`
   连接地址，按 API 类型决定是否必填。
6. `APIKeyCiphertext string`
   加密后的密钥，不回传前端明文。
7. `Avatar string`
   提供商头像，可存 URL、资源路径或图标标识。
8. `ExtraConfigJSON string`
   预留 API 级特殊参数，如 region、organization、project 等。
9. `Enabled bool`
    是否可用。
10. `CreatedAt / UpdatedAt / DeletedAt`
    与现有模型保持一致。

建议新增模型 `ModelCatalog`，字段如下：

1. `ID uint`
   内部自增主键。
2. `ModelID string`
   对外业务 ID，UUID。
3. `ProviderID string`
   关联 `ModelProvider.ProviderID`。
4. `ModelKey string`
   实际请求使用的模型标识，如 `deepseek-chat`。
5. `Name string`
   展示名称。
6. `Avatar string`
   模型头像。
7. `CapabilitiesJSON string`
   预留能力标签，如视觉、函数调用、推理等。
8. `Enabled bool`
   是否可选。
9. `Sort int`
   前端展示排序。
10. `CreatedAt / UpdatedAt / DeletedAt`
    与现有模型保持一致。

建议新增单独的默认配置模型 `DefaultModelConfig`，字段如下：

1. `ID uint`
   内部主键。
2. `ConfigKey string`
   固定值，如 `system_default_chat_model`。
3. `ModelID string`
   当前系统默认模型，对应 `ModelCatalog.ModelID`。
4. `UpdatedAt`
   最近更新时间。

说明：

1. `api_key`、`base_url` 严格放在提供商维度。
2. 提供商和模型各自维护独立头像字段。
3. 默认模型单独配置，不挂在提供商表或模型表的布尔字段上。

### 4.4 前端展示所需的“扁平模型列表”

聊天页不应该直接理解全部 provider 配置，建议后端提供一个扁平化列表给前端：

```json
[
  {
    "provider_id": "9b08f9d2-9c8f-4b5b-b8fb-4c0b9d65d4f2",
    "provider_name": "默认 DeepSeek 网关",
    "provider_avatar": "deepseek-provider.png",
    "api_type": "openai_compatible",
    "model_id": "6fef6f48-4c0b-45b2-9831-cff6b708ff74",
    "model_key": "deepseek-chat",
    "model_name": "DeepSeek Chat",
    "model_avatar": "deepseek-chat.png",
    "is_system_default": true
  }
]
```

这样前端只需要做展示、选择和回传，不需要拼接 provider 参数，但依然能展示提供商和模型两个维度的信息。

## 5. 前端修改方案

### 5.1 快捷入口增加“模型服务”卡片

修改位置：

1. `openIntern_forentend/app/(workspace)/layout.tsx`

方案：

1. 在“快捷入口”区块新增一个与 `A2UI 管理` / `Skill 市场` 同风格的按钮。
2. 文案建议使用：`模型服务`
3. 跳转目标建议新增为：`/models`

原因：

1. 与现有 `/a2ui`、`/skills`、`/kb` 路由风格一致。
2. 路径短，后续也便于扩展详情页或子页面。

### 5.2 新增模型服务管理页

建议新增：

1. `openIntern_forentend/app/(workspace)/models/page.tsx`

页面职责：

1. 展示模型提供商列表。
2. 展示模型目录列表。
3. 新增/编辑/删除模型提供商配置。
4. 新增/编辑/删除模型目录项。
5. 单独设置系统默认模型。
6. 对密钥字段只显示掩码，不回显明文。

建议首版表单字段：

1. 提供商表单：
   名称、API 类型、头像、base_url、api_key、extra_config_json、启用状态
2. 模型表单：
   所属提供商、模型标识、展示名称、头像、capabilities_json、启用状态、排序
3. 默认模型设置：
   单独选择一个全局默认 `ModelID`

说明：

1. 本页的交互风格可直接复用现有 `a2ui` / `skills` 页的卡片布局。
2. 不必首版做“自动拉取厂商模型列表”，先用手工配置更稳妥。

### 5.3 聊天页增加模式选择

修改位置：

1. `openIntern_forentend/app/(workspace)/chat/page.semi.tsx`

新增前端状态：

1. `conversationMode: "chat" | "agent"`
2. `selectedProviderId: string`
3. `selectedModelId: string`
4. `availableModels: Array<...>`

实现方式：

1. 使用 `AIChatInput.renderConfigureArea`
2. 在输入框底部渲染一个上拉选择控件，文案为：
   `Chat 模式`、`Agent 模式`

行为约定：

1. 默认选中 `Chat 模式`
2. 切到 `Agent 模式` 时：
   继续显示占位 UI，但发送时直接提示“Agent 模式暂未开放”
3. 切回 `Chat 模式` 时：
   恢复模型选择按钮与正常发送

### 5.4 聊天页增加模型选择

修改位置同上，仍在 `page.semi.tsx`。

实现方式：

1. 使用 `AIChatInput.renderActionArea`
2. 仅在 `conversationMode === "chat"` 时显示
3. 动作区放置一个“模型选择”按钮，点击后展示下拉列表或弹出菜单

数据来源：

1. 页面加载时调用后端“模型目录聚合”接口
2. 后端返回扁平模型列表
3. 前端以“提供商 + 模型”展示
4. 在选择项中显示提供商头像和模型头像

交互约定：

1. 如果后端标记了系统默认模型，则自动选中
2. 如果没有任何可用模型：
   输入框继续可见，但发送前提示“请先配置模型服务”

### 5.5 将选择结果透传给后端

当前 `handleMessageSend` 中是：

1. `agent.setMessages([message])`
2. `agent.runAgent()`

应改为：

1. 保持 `agent.setMessages([message])` 不变
2. 调用 `agent.runAgent({ forwardedProps: ... })`

推荐透传：

```ts
agent.runAgent({
  forwardedProps: {
    agentConfig: {
      conversation: {
        mode: conversationMode,
      },
      model:
        conversationMode === "chat" && selectedModelId
          ? {
              modelId: selectedModelId,
              providerId: selectedProviderId || undefined,
            }
          : undefined,
      features: {},
    },
  },
});
```

说明：

1. 这与当前项目文档中已记录的 `runAgent(parameters?: RunAgentParameters)` 用法一致。
2. 发送内容仍旧只走增量消息，和当前基于 AG-UI 事件订阅的实现不冲突。

## 6. 后端修改方案

### 6.1 新增模型服务数据模型

建议新增文件：

1. `openIntern_backend/internal/models/model_provider.go`
2. `openIntern_backend/internal/models/model_catalog.go`
3. `openIntern_backend/internal/models/default_model_config.go`
4. `openIntern_backend/internal/services/model_provider.go`
5. `openIntern_backend/internal/services/model_catalog.go`
6. `openIntern_backend/internal/controllers/model_provider.go`
7. `openIntern_backend/internal/controllers/model_catalog.go`

并在 `internal/database/database.go` 的 `AutoMigrate` 中加入上述三个模型。

### 6.2 新增模型服务管理 API

建议拆成三组 API：

1. 提供商管理：
   `GET /v1/model-providers`
   `POST /v1/model-providers`
   `GET /v1/model-providers/:id`
   `PUT /v1/model-providers/:id`
   `DELETE /v1/model-providers/:id`
2. 模型目录管理：
   `GET /v1/models`
   `POST /v1/models`
   `GET /v1/models/:id`
   `PUT /v1/models/:id`
   `DELETE /v1/models/:id`
3. 默认模型与聊天页聚合：
   `GET /v1/models/default`
   `PUT /v1/models/default`
   `GET /v1/models/catalog`

可选增强（首版可不做）：

1. `POST /v1/models/:id/validate`
   测试模型可用性
2. `POST /v1/model-providers/:id/validate`
   测试提供商连接
3. `POST /v1/model-providers/:id/discover`
   尝试从提供商拉取模型列表

### 6.3 扩展 `forwardedProps` 处理链

当前 `applyForwardedPropsChain` 只处理 `a2uiAction`。

建议新增一个总入口解析器：

1. `agentConfigForwardedPropsHandler`

该 handler 只负责读取 `forwardedProps.agentConfig`，并在内部按层级解析 `conversation`、`model`、`features`。

不建议再把“模型选择”转换成用户消息，而应做“运行时配置解析”。

推荐改造方向：

1. 将 `forwardedPropsChainContext` 扩展为可携带 `runtimeConfig`
2. 在 handler 中把：
   `mode`、`providerId`、`modelId`
   解析到统一运行时结构
3. 后续 `RunAgent` 直接使用这个结构体进行模型路由

建议新增结构：

```go
type AgentRuntimeConfig struct {
    Conversation struct {
        Mode string
    }
    Model struct {
        ProviderID string
        ModelID    string
    }
    Features map[string]any
}
```

### 6.4 将“固定单例模型”改为“按请求解析模型”

当前问题：

1. `InitEino` 在启动时构造固定 `chatModel`
2. `adk.NewChatModelAgent` 和 `einoRunner` 也在启动时固定下来

这不适合按请求切换模型，也不适合按 `APIType` 动态切换装配器。

推荐方案：

1. 保留“工具、中间件、标题模型、全局配置”在启动时初始化
2. 把“主对话 ChatModel”改成按请求构造
3. `RunAgent` 内在解析完 `forwardedProps` 后：
   先读取默认模型配置或指定模型，再联表读取对应提供商，最后构造 `ChatModel`
4. 再基于该 `ChatModel` 临时构建本次请求使用的 `adk.ChatModelAgent` 和 `Runner`

即：

1. 全局缓存“工具列表”和“技能中间件”
2. 每次请求只新建“模型相关对象”

这样改动最小，也最符合当前代码结构。

### 6.5 Eino `ChatModel` 装配方式

建议新增统一工厂，例如：

1. `buildChatModel(ctx, providerConfig, modelConfig) (schema.ToolCallingChatModel, error)`

工厂职责：

1. 根据 `APIType` 做分发
2. 将提供商表中的连接信息和模型表中的模型标识组合起来
3. 将数据库中的字段映射到对应 Eino 配置
4. 如果用户未显式选择模型，则回退到系统默认模型

映射建议：

1. `ark`
   使用一套 Ark 装配逻辑，所有归属 `ark` 的提供商都走这一套
2. `openai_compatible`
   使用一套 OpenAI 兼容装配逻辑，所有兼容提供商都走这一套
3. `anthropic`
   使用一套 Anthropic 装配逻辑
4. `gemini`
   使用一套 Gemini 装配逻辑
5. `ollama`
   使用一套 Ollama 装配逻辑

注意：

1. 具体 Eino 扩展包以最终落地时引入的官方组件为准。
2. 代码维护对象应是“少量 API 类型装配器”，不是“每个 provider 一套硬编码”。

### 6.6 默认路由与兜底规则

建议后端统一执行以下规则：

1. 如果前端未传 `agentConfig.model.modelId`
   使用 `DefaultModelConfig` 中配置的系统默认模型
2. 如果传了 `modelId`
   以 `modelId` 为主查询模型，再反查所属提供商
3. 如果同时传了 `providerId`
   只作为一致性校验，不作为默认回退来源
4. 如果传入的提供商不存在或被禁用
   返回明确错误，不要静默回退
5. 如果模型不存在、被禁用，或不属于指定提供商
   返回参数错误
6. 如果 `agentConfig.conversation.mode=agent`
   当前阶段直接返回“暂未开放”或在前端截断，不进入实际推理链路

### 6.7 安全要求

模型服务管理涉及密钥，建议把这部分写成硬约束：

1. API key 在数据库中加密存储，不明文回传前端
2. 列表接口只返回 `api_key_masked`
3. 更新接口中，如果 `api_key` 为空，表示“不变更原密钥”
4. 只有通过鉴权的用户才能访问模型服务接口
5. 如果暂时不做细粒度权限，首版先按“全局配置”处理，不做用户级隔离

## 7. 推荐的实施顺序

按风险从低到高，建议这样拆分：

### 第一阶段：基础数据与管理能力

1. 新增 `ModelProvider`、`ModelCatalog`、`DefaultModelConfig` 三个模型
2. 完成 `/v1/model-providers`、`/v1/models`、`/v1/models/default`、`/v1/models/catalog`
3. 新增前端 `/models` 管理页
4. 在侧边栏增加“模型服务”快捷入口

这一阶段完成后，系统已经可以“配置模型”，但聊天页还不切换模型。

### 第二阶段：聊天页 UI 接入

1. 在 `AIChatInput` 上增加 `renderConfigureArea`
2. 在 `AIChatInput` 上增加 `renderActionArea`
3. 聊天页加载 `/v1/models/catalog`
4. `handleMessageSend` 通过 `agent.runAgent({ forwardedProps })` 传递 `agentConfig`

这一阶段完成后，前端已经能把选择结果送到后端。

### 第三阶段：后端动态模型路由

1. 扩展 `forwardedProps` 解析
2. 把固定 `einoRunner` 改为“工具全局缓存 + 模型按请求构造”
3. 新增按 `APIType` 分发的 `buildChatModel` 工厂
4. 完成默认模型、参数校验、错误返回

这一阶段完成后，聊天才能真正按所选模型运行。

### 第四阶段：增强能力（可选）

1. 连接测试
2. 自动探测模型列表
3. API 类型模板与 provider 参数模板
4. 用户级默认模型偏好

## 8. 风险与注意事项

1. 如果继续沿用全局 `einoRunner`，模型切换需求会被架构卡死，这是本次改造的核心技术点。
2. `forwardedProps` 当前已有 `a2uiAction` 约定，新功能不应继续堆顶层散键，建议统一收敛到 `agentConfig`。
3. 如果前端直接把完整 provider 配置塞进 `forwardedProps`，会把密钥暴露到浏览器和日志里，必须避免。
4. 提供商和模型拆表后，删除提供商前需要处理关联模型，避免脏数据。
5. 首版不建议同时做“自动发现 provider 模型列表”，因为不同 provider 差异明显，先用手工配置能降低不确定性。
6. `agent` 模式既然本期只是占位，前后端都要明确标注，避免给用户造成“已可用”的误导。

## 9. 建议的首版验收标准

满足以下条件即可认为首版方案可进入开发：

1. 左侧快捷入口可进入“模型服务”管理页
2. 可新增模型提供商和模型目录项，并在列表中查看
3. 聊天页底部可切换 `Chat 模式 / Agent 模式`
4. `Chat 模式` 下可选择模型
5. 发送消息时，后端能从 `forwardedProps.agentConfig` 解析出模型选择
6. 后端能根据 `APIType + ProviderConfig + ModelConfig` 成功构造对应 Eino `ChatModel`
7. 对话实际按选中模型执行

## 11. 建议的后续实现边界

4. 等首轮闭环稳定后，再补更多 provider 和高级参数
