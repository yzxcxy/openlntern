# 聊天多模态功能开发方案（调研版）

## 1. 目标与范围

目标：在聊天窗口支持上传图片/文件/音频/视频，上传后拿到后端返回链接；发送时把文本与多模态链接放进**同一条 AGUI user message**；后端将其转换为 Eino 的多模态消息体参与推理。

本方案只输出设计与实施清单，不涉及本次代码开发。

## 2. 当前链路调研结论

### 2.1 前端现状

1. 聊天发送只支持纯文本：`handleMessageSend` 里仅提取 `text`，并构造 `AguiMessage.content: string`。
2. 上传按钮已关闭：`AIChatInput` 使用 `showUploadButton={false}`、`showUploadFile={false}`。
3. 历史消息映射中，user 侧只取文本：`mapHistoryMessages` 对 `role=user` 调用 `extractText`，不会渲染附件卡片。

### 2.2 后端现状

1. `/v1/chat/sse` 已接收 `RunAgentInput.messages` 并走 AGUI -> Eino 转换。
2. 聊天场景暂无通用上传接口（当前只存在头像、插件图标、知识库文件等上传）。
3. 已有 COS 上传基础能力：`services.File.UploadWithKey`/`UploadReader`。

### 2.3 AGUI -> Eino 转换能力

当前后端已具备 AGUI -> Eino 的转换主链路，但用户多模态输入 part 解析仍需与 AGUI 官方 `binary` 结构对齐（本次实施项）。

## 3. 目标链路（实施后）

1. 用户在聊天输入区选择本地文件（图片/音频/视频/其他文件）。
2. 前端立即调用后端上传接口，得到公网 URL 与文件元数据。
3. 上传项在输入框上方展示“待发送附件”。
4. 用户点击发送时：
   - 构造一条 AGUI `role=user` 消息；
   - `content` 改为数组，文本与附件 URL 一起放入；
   - 通过 `agent.setMessages([message])` + `runAgent(...)` 发起。
5. 后端通过 `AGUIRunInputToEinoMessages` 转换成 Eino 多模态消息。

## 4. 协议与数据结构设计

### 4.1 新增上传接口（后端）

- Method: `POST /v1/chat/uploads`
- Auth: 复用 `middleware.AuthRequired()`
- Content-Type: `multipart/form-data`
- FormData:
  - `file`: 文件本体（必填）
  - `thread_id`: 会话 ID（可选，便于对象路径归档）

返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "key": "chat/{user_id}/{thread_id}/{yyyyMMdd}/{uuid}.ext",
    "url": "https://xxx.cos.xxx/...",
    "mime_type": "image/png",
    "file_name": "demo.png",
    "size": 12345,
    "media_kind": "image"
  }
}
```

`media_kind` 规则：
- `image/*` -> `image`
- `audio/*` -> `audio`
- `video/*` -> `video`
- 其他 -> `file`

### 4.2 AGUI user message（关键位置约定）

发送时必须放在：`run_input.messages[最后一条].content`。

不能放在：
- `forwardedProps`
- `context`
- 单独新建另一条 message

统一结构示例（按 AGUI 官方 user message）：

```json
{
  "id": "msg-user-1",
  "role": "user",
  "content": [
    { "type": "text", "text": "请根据附件总结重点" },
    {
      "type": "binary",
      "mimeType": "image/png",
      "url": "https://.../a.png",
      "filename": "a.png"
    },
    {
      "type": "binary",
      "mimeType": "application/pdf",
      "url": "https://.../b.pdf",
      "filename": "b.pdf"
    }
  ]
}
```

位置与顺序约定：
1. 文本 part 建议放在 `content[0]`（若文本为空可省略）。
2. 上传文件 part 按用户选择顺序依次追加在后。

## 5. 前端改造方案（`chat/page.semi.tsx`）

### 5.1 状态与类型

新增上传态：
- `pendingUploads: UploadAssetItem[]`（已上传、待发送）
- `uploading: boolean`
- `uploadError: string`

`UploadAssetItem` 建议字段：
- `id`, `url`, `key`, `fileName`, `mimeType`, `size`, `mediaKind`

### 5.2 交互流程

1. 在输入区增加上传入口（按钮或自定义 `input[type=file]`）。
2. 文件选中后立即调用 `/api/backend/v1/chat/uploads`。
3. 成功后写入 `pendingUploads`，在输入框上方显示预览项（可删除）。
4. 点击发送：
   - 允许“纯附件无文本”发送（至少文本/附件之一存在）；
   - 构造 AGUI `content[]`；
   - 本地回显为 Semi `input_*` 内容项（图片/文件 + 文本；音频/视频以文件项展示）；
   - 发送后清空 `pendingUploads`。

### 5.3 历史消息映射

`mapHistoryMessages` 的 user 分支从“仅文本”改为：
- 若 `content` 是字符串：按旧逻辑渲染文本。
- 若 `content` 是数组：解析并映射为 Semi 的 `input_text/input_image/input_file/...` 结构，保证历史可回显附件。

## 6. 后端改造方案

### 6.1 路由与控制器

新增：
- 路由：`POST /v1/chat/uploads`
- 控制器：`controllers.UploadChatAsset`

职责：
1. 读取 `multipart` 文件。
2. 校验文件非空、MIME 类型、大小上限。
3. 生成 COS key（按 user/thread/date 分层）。
4. 调用 `services.File.UploadWithKey`。
5. 返回 URL + 元数据。

### 6.2 服务层

建议新增 `services/chat_upload.go`：
- 封装 key 生成、MIME -> media_kind 判定、上传统一错误处理。

### 6.3 Agent 主链路

需要补充转换逻辑：
- 将 AGUI `InputContent[]` 中 `type=binary` 按 `mimeType` 映射为 Eino 输入 part：
  - `image/*` -> `image_url`
  - `audio/*` -> `audio_url`
  - `video/*` -> `video_url`
  - 其他 -> `file_url`
- 持久化逻辑无需额外改造，继续保存 user 多模态消息 JSON。

## 7. 验收标准（DoD）

1. 聊天页可上传图片/文件/音频/视频，并显示在输入区上方。
2. 发送后请求体中，`messages` 最后一条 user message 的 `content` 含文本+上传 URL part。
3. 后端能把该消息正确转换到 Eino `UserInputMultiContent`。
4. 模型能基于附件进行回答（至少图片+PDF 各 1 个用例）。
5. 历史消息刷新后仍能正确渲染已发送附件。
6. 失败路径可用：上传失败、发送失败、超限/非法 MIME 提示明确。

## 8. 分阶段实施建议

### Phase 1（后端能力）

- 新增聊天上传 API + 基本校验 + 响应结构。

### Phase 2（前端上传与发送）

- 输入区上传、预览、删除、发送组包。

### Phase 3（历史与渲染）

- user 多模态历史回显与样式完善。

### Phase 4（联调与回归）

- 覆盖文本+图片、文本+文件、纯附件、多附件混合。

## 9. 风险与注意事项

1. 前端发送必须统一为 AGUI 官方 `binary`，禁止继续发送非官方 `image_url/file_url` 用户输入结构。
2. 文件大小与 MIME 白名单需在后端强约束，前端仅做体验层校验。
3. COS 未配置时上传接口应返回明确错误，避免前端静默失败。
4. 建议限制同次发送附件数量（例如 10）和单文件大小（例如 20MB，可配置）。
