# Agent 市场与多智能体能力开发方案（调研版）

> 状态：仅方案设计，不包含本次代码实现。  
> 日期：2026-03-22

## 1. 目标与范围

目标：在当前 openIntern 的聊天底座上，增加“可配置 agent 实体”和“Agent 市场”，支持：

1. Agent 市场列表、创建、启用、编辑、测试、修改。
2. Agent 类型先支持两类：
   - `single_agent`
   - `supervisor_agent`
3. `single_agent` 不能绑定其他 agent。
4. `supervisor_agent` 除主 agent 自身资源外，还能绑定 subagent。
5. single/supervisor 都支持配置：
   - 系统提示词
   - 智能体名字
   - 智能体描述
   - 示例问法
   - 聊天背景
   - agent 头像
   - 默认模型
   - 工具
   - skill
   - 知识库
   - 是否开启 agent 级长期记忆
6. 创建页和编辑页必须内置“测试”能力。

本方案优先追求两点：

1. 最大化复用现有 `RunAgent`、模型、工具、skill、知识库和 SSE 聊天链路。
2. 不额外造第二套聊天系统，只把“agent 作为一等配置对象”接到现有系统里。

## 2. 现状调研结论

### 2.1 后端现状

当前后端已经有一个可工作的通用 agent runtime，但还没有“agent 配置实体”。

已有能力：

1. `internal/services/agent` 已基于 Eino ADK + `ChatModelAgent` 跑通 SSE 聊天。
2. 模型服务已具备：
   - `model_provider`
   - `model_catalog`
   - 全局默认模型 `default_model_config`
3. 插件工具已具备：
   - 插件/工具 CRUD
   - 运行时装配
   - tool search 模式基础
4. Skill 已具备：
   - 远端存储
   - frontmatter 元数据
   - Eino skill middleware
5. 知识库已具备：
   - 列表、导入、树查看、检索注入
6. 长期记忆已具备基础版：
   - 检索注入
   - thread 级异步 sync
   - usage log

关键缺口：

1. `internal/models` 与 `database.AutoMigrate` 中没有 `agent` 相关表。
2. `RunAgent` 仍然是“通用聊天入口”，没有“加载某个 agent 定义后运行”的分支。
3. 前端聊天页虽然有 `conversationMode: "chat" | "agent"`，但当前被硬编码为不可用：
   - 前端：`Agent 模式暂未开放`
   - 后端：`agent mode is not available yet`
4. Sidebar 当前没有 Agent 市场入口。

### 2.2 运行链路现状

当前聊天运行顺序大致是：

1. 读取线程历史。
2. 解析 `forwardedProps.agentConfig`。
3. 解析 `forwardedProps.contextSelections`。
4. 注入知识库检索结果与 skill 指令。
5. 注入长期记忆。
6. 做上下文压缩。
7. 组装 `ChatModelAgent` + 工具 + middleware。
8. 流式输出并持久化消息。

这说明当前系统已经具备：

1. 资源注入能力。
2. 运行时模型切换能力。
3. 工具动态装配能力。
4. SSE 测试/聊天能力。

因此 Agent 市场最合理的方案不是再造一套 Runner，而是：

1. 新增 agent 配置存储；
2. 在运行前把 agent 配置解析成现有 runtime 可以理解的“资源集合”和“执行配置”；
3. 让 `single_agent` / `supervisor_agent` 都落到现有 `ChatModelAgent` 模式上。

### 2.3 为什么不建议直接切 DeepAgent

当前仓库已经围绕以下模式搭好了完整链路：

1. `ChatModelAgent`
2. 自定义工具装配
3. skill middleware
4. 上下文压缩
5. AG-UI SSE 输出

如果此时强切 `DeepAgent`，会额外引入：

1. 子 agent 管理方式变化；
2. 工具与上下文注入方式变化；
3. 现有消息持久化与事件流适配成本；
4. 调试复杂度增加。

所以本期建议：

1. `single_agent` 使用 `ChatModelAgent`
2. `supervisor_agent` 仍使用 `ChatModelAgent`
3. subagent 通过 `adk.NewAgentTool(...)` 封装为父 agent 的工具

这样最贴合当前代码现状，也最容易分阶段落地。

## 3. 目标方案总览

### 3.1 核心思路

新增一层 `AgentDefinition`：

1. Agent 市场负责管理 `AgentDefinition`
2. 运行时负责把 `AgentDefinition` 编译成 `RuntimeAgentPlan`
3. `RuntimeAgentPlan` 再交给现有 `RunAgent`/Runner 执行

### 3.2 两类 agent 的运行方式

#### single_agent

运行时表现为一个配置化的 `ChatModelAgent`：

1. `Instruction` 来自系统提示词
2. 模型来自 agent 默认模型
3. 工具来自绑定的工具列表
4. skill / 知识库 / 长期记忆在消息进入模型前注入

#### supervisor_agent

运行时表现为一个“主 ChatModelAgent + subagent tools”：

1. 主 agent 仍然有自己的系统提示词、模型、工具、skill、知识库、长期记忆
2. 每个 subagent 先被构造成独立 agent
3. 再通过 `adk.NewAgentTool` 封装成主 agent 可调用工具
4. 主 agent 决定是否把任务转交给 subagent

## 4. 数据模型设计

### 4.1 推荐表结构

#### `agent`

主表，负责存基础配置。

建议字段：

1. `agent_id`：业务主键，uuid
2. `name`
3. `description`
4. `agent_type`：`single_agent` / `supervisor_agent`
5. `status`：`draft` / `enabled` / `disabled`
6. `avatar`
7. `chat_background_json`
8. `system_prompt`
9. `example_questions_json`
10. `default_model_id`
11. `long_term_memory_enabled`
12. `version`
13. `last_tested_at`
14. `created_at`
15. `updated_at`
16. `deleted_at`

说明：

1. `example_questions_json` 用 JSON 数组，避免为了 3~5 条示例问法单独建表。
2. `chat_background_json` 建议直接用 JSON，不要先设计成单一字符串。
   - MVP 前端只暴露“背景图 URL / 预设背景 key”也没关系。
   - 但存储层应预留扩展空间。
3. `status` 需要有 `draft`，因为“创建页内测试”天然要求未启用也能运行。

#### `agent_tool_binding`

1. `agent_id`
2. `tool_id`
3. `sort`
4. `created_at`

#### `agent_skill_binding`

1. `agent_id`
2. `skill_name`
3. `sort`
4. `created_at`

#### `agent_kb_binding`

1. `agent_id`
2. `kb_name`
3. `sort`
4. `created_at`

#### `agent_subagent_binding`

1. `agent_id`
2. `subagent_id`
3. `sort`
4. `created_at`

约束：

1. `single_agent` 不允许出现任何 `agent_subagent_binding`
2. `agent_id != subagent_id`
3. 不允许形成环

### 4.2 为什么绑定表建议拆开

不建议把工具、skill、知识库、subagent 全塞进 `agent.resources_json`，原因：

1. 当前仓库资源都是独立实体，拆表更利于校验。
2. 绑定关系查询频繁，拆表更容易做启用态过滤。
3. supervisor 的 subagent 关系需要做环检测，拆表更清晰。
4. 后续若增加统计、影响分析、引用检查，拆表更省事。

### 4.3 与现有模型表的关系

1. `default_model_id -> model_catalog.model_id`
2. `tool_id -> tool.tool_id`
3. `skill_name -> skill_frontmatter.skill_name`
4. `kb_name -> openviking resources 根目录下知识库名`
5. `subagent_id -> agent.agent_id`

## 5. 状态机与约束设计

### 5.1 Agent 状态

建议三态：

1. `draft`
2. `enabled`
3. `disabled`

规则：

1. 新建默认 `draft`
2. `draft` 可以测试、编辑，但不出现在市场“可用 agent”列表，也不能被绑定为 subagent
3. `enabled` 可以被聊天页选择，也可以被 supervisor 绑定
4. `disabled` 可以编辑，但不能运行，也不能被绑定

### 5.2 启用校验

从 `draft/disabled -> enabled` 时，后端必须做一次完整校验：

1. `name` 非空
2. `agent_type` 合法
3. `default_model_id` 存在且模型、provider 都启用
4. 所有工具都必须是启用态
5. 所有 skill 必须存在
6. 所有知识库必须存在
7. 若是 supervisor：
   - subagent 全部为 `enabled`
   - 不存在自引用
   - 不存在环

### 5.3 禁用校验

建议增加一条保护：

1. 若某个 agent 正被 `enabled` 的 supervisor 引用，则禁止直接禁用
2. 返回明确错误，提示先解除引用或先禁用 supervisor

否则很容易出现：

1. 市场列表上看 supervisor 是启用态
2. 但运行时某个 subagent 已失效
3. 最终错误被延迟到聊天阶段

## 6. 运行时设计

### 6.1 新增运行上下文

在现有 `AgentRuntimeConfig` 的基础上，新增 agent 维度配置，例如：

```json
{
  "conversation": {
    "mode": "agent",
    "agentId": "xxx"
  },
  "test": {
    "enabled": true
  }
}
```

同时支持一个“草稿测试”输入结构，用于创建/编辑页：

```json
{
  "agentDraft": {
    "...": "未保存的 agent 定义"
  },
  "messages": [
    { "role": "user", "content": "..." }
  ]
}
```

### 6.2 运行前编译步骤

新增 `CompileAgentDefinition(...)`：

1. 加载 agent 基础信息
2. 加载工具/skill/kb/subagent 绑定
3. 做启用态与存在性校验
4. 编译成 `RuntimeAgentPlan`

`RuntimeAgentPlan` 建议包含：

1. 基础信息：`agent_id`、`agent_type`、`name`
2. 模型选择：`provider_id`、`model_id`
3. 提示词：`system_prompt`
4. 资源绑定：
   - `tool_ids`
   - `skill_names`
   - `kb_names`
5. `long_term_memory_enabled`
6. `subagents`

### 6.3 single_agent 的组装

在当前 `buildEinoRunner(...)` 基础上新增 agent 分支：

1. `Instruction` 使用 agent 的系统提示词
2. 工具装配不再走前端自由选择，而是走 agent 绑定
3. 技能和知识库在 forwarded props 处理前后统一注入
4. 模型优先使用 agent 默认模型

推荐顺序：

1. 读取线程历史
2. 读取 agent 定义
3. 合并 agent 绑定资源
4. 注入知识库命中结果
5. 注入 skill 指令
6. 注入长期记忆
7. 上下文压缩
8. 创建 `ChatModelAgent`

### 6.4 supervisor_agent 的组装

`supervisor_agent` 的核心是“主 agent + subagent tool 化”。

建议实现步骤：

1. 先递归编译每个 subagent 的 `RuntimeAgentPlan`
2. 将 subagent 构造成独立 `ChatModelAgent`
3. 使用 `adk.NewAgentTool(ctx, subAgent)` 封装成主 agent 可调用工具
4. 把这些 subagent tools 与主 agent 自身工具合并
5. 用主 agent 的系统提示词创建最终 `ChatModelAgent`

运行时约束：

1. 需要做环检测
2. 需要做深度限制，建议 MVP 限制最大深度 `3`
3. 需要做去重，避免 supervisor 绑定重复 subagent

### 6.5 为什么 subagent 适合做成 tool

原因：

1. 和当前 `ChatModelAgent` 模型最兼容
2. subagent 可天然获得独立上下文
3. 主 agent 不必共享 subagent 的全部消息历史
4. 很适合“调度者 -> 专家 agent”的场景

## 7. 资源注入与提示词规则

### 7.1 系统提示词

当前 `ChatModelAgentConfig` 里还没有真正使用 `Instruction`。

本方案建议：

1. `system_prompt` 直接映射到 `Instruction`
2. 若为空则不传
3. supervisor 与 single 都支持

### 7.2 skill 注入

当前 skill 机制是：

1. 通过 skill middleware 注册 skill 能力
2. 通过临时消息提示模型优先参考某些 skill

本方案建议复用现有机制，不新增第二套：

1. agent 绑定的 `skill_names` 直接转成 `contextSelections.skills`
2. 继续复用当前 `buildSkillInstructionMessage(...)`

### 7.3 知识库注入

继续复用当前机制：

1. 根据最后一条 user message 做检索
2. 命中内容作为临时 system message 注入

区别只在于：

1. 过去由前端 mention 选择
2. 现在还要叠加 agent 绑定的知识库

### 7.4 工具注入

agent 绑定工具时，建议只存 `tool_id`，运行时再解析为：

1. code tools
2. api tools
3. mcp tools

仍复用现有 `pluginsvc.Plugin.Build...RuntimeTools(...)` 能力，不自己重造。

### 7.5 聊天背景

建议明确为“前端展示字段”，不直接进入 prompt。

原因：

1. 它更像 UI 配置，不是模型能力配置
2. 和系统提示词混用容易造成职责不清

MVP 可以先支持：

1. 背景图 URL
2. 预设背景 key

但底层建议仍存 `chat_background_json`。

## 8. 长期记忆设计

### 8.1 当前问题

当前长期记忆是 thread 语义：

1. 检索默认从固定根：
   - `viking://user/default/memories/`
   - `viking://agent/default/memories/`
2. sync 状态也按 `thread_id` 管理

这和“agent 级长期记忆”并不完全一致。

### 8.2 目标语义

本需求里的“agent 级长期记忆”建议解释为：

1. 某个 agent 自己运行过程中形成的长期经验
2. 被这个 agent 后续运行复用
3. 不应该和所有 agent 共用同一个 `default agent memory root`

### 8.3 推荐改造

新增 `MemoryQueryOptions` / `MemorySyncScope`：

1. `agent_id`
2. `thread_id`
3. `enable_agent_memory`

OpenViking 目标根建议改成按 agent 分桶：

1. `viking://agent/openintern/agents/{agent_id}/memories/`

同步时：

1. thread 仍然是会话维度
2. 但 sync state 需要额外带上 `agent_id`
3. session id 不建议继续只用 `thread_id`
4. 建议改为：`agent:{agent_id}:thread:{thread_id}`

### 8.4 表结构建议

可以二选一：

#### 方案 A：改造现有 `memory_sync_state`

新增字段：

1. `agent_id`
2. `scope_type`
3. `scope_key`

优点：

1. 复用现有 worker

缺点：

1. 需要较多兼容逻辑

#### 方案 B：新增 `agent_memory_sync_state`

优点：

1. 边界清晰
2. 不会污染现有通用 chat memory 行为

缺点：

1. 需要双 worker 或统一抽象

本项目更建议 **方案 A**，但要把“通用聊天”和“agent 运行”都统一抽象为 `MemoryScope`，不要再在上层写死 thread-only 语义。

## 9. API 设计

### 9.1 Agent 市场 CRUD

建议新增：

1. `GET /v1/agents`
2. `POST /v1/agents`
3. `GET /v1/agents/:id`
4. `PUT /v1/agents/:id`
5. `DELETE /v1/agents/:id`
6. `POST /v1/agents/:id/enable`
7. `POST /v1/agents/:id/disable`
8. `GET /v1/agents/enabled-options`

`enabled-options` 用于：

1. supervisor 绑定 subagent 下拉框
2. 聊天页 agent 模式选择器

### 9.2 测试接口

创建/编辑页必须能测试未保存草稿，所以必须有独立测试接口。

建议新增：

1. `POST /v1/agents/debug/sse`

请求体：

1. `agent_draft`
2. `messages`
3. `thread_id`（测试专用，可选）

行为要求：

1. 允许测试 `draft`
2. 不写入正式 `thread` / `message`
3. 不触发 thread 标题生成
4. 不进入正式历史记录
5. 不触发正式长期记忆 sync

这是创建页/编辑页测试能力的关键点。

### 9.3 正式聊天接口

不建议再新造一条完整聊天协议，建议复用现有 `/v1/chat/sse`，只扩展 agent 配置：

```json
{
  "forwardedProps": {
    "agentConfig": {
      "conversation": {
        "mode": "agent",
        "agentId": "xxx"
      }
    }
  }
}
```

这样收益最大：

1. 复用现有 AG-UI 客户端
2. 复用现有线程历史
3. 复用现有 SSE 渲染

## 10. 前端方案

### 10.1 新增页面

建议新增：

1. `/agents`
2. `/agents/new`
3. `/agents/[id]`

Sidebar 新增“Agent 市场”入口。

### 10.2 市场页

列表页支持：

1. 搜索
2. 按类型筛选
3. 按状态筛选
4. 卡片展示：
   - 头像
   - 名字
   - 描述
   - 类型
   - 状态
   - 默认模型
   - subagent 数量
5. 操作：
   - 创建
   - 编辑
   - 启用/禁用
   - 测试
   - 删除

### 10.3 创建/编辑页

推荐用“左表单 + 右测试面板”的双栏结构。

左侧表单分区：

1. 基础信息
   - 名字
   - 描述
   - 类型
   - 头像
   - 聊天背景
2. Prompt 与体验
   - 系统提示词
   - 示例问法
3. 运行配置
   - 默认模型
   - 长期记忆开关
4. 资源绑定
   - 工具
   - skill
   - 知识库
5. supervisor 专区
   - subagent 选择

右侧测试面板：

1. 复用现有聊天消息渲染
2. 复用现有输入框
3. 点击发送时调用 `/v1/agents/debug/sse`
4. 测试记录仅在当前页面内存存在，不进正式历史

### 10.4 聊天页如何接入 agent

建议分两期：

#### 一期

1. 先完成市场页内测试能力
2. 聊天主页面暂不开放 agent mode

#### 二期

1. 聊天页顶部的 `conversationMode=agent` 真正开放
2. 用户选择某个 enabled agent
3. 隐藏自由模型/工具选择，改为显示“当前使用的 agent 配置”

这样可以把风险拆开，不把“市场能力”和“主聊天入口改造”绑死在一个迭代里。

## 12. 关键校验逻辑

### 12.1 保存时校验

4. 工具是否都存在
5. skill 是否都存在
6. 知识库是否都存在

### 12.2 启用时校验

1. 所有资源必须可运行
2. subagent 必须全部 enabled
3. 必须无环
4. 必须满足深度限制

### 12.3 运行时兜底校验

即便保存/启用时做过校验，运行前仍要再校验一次：

1. 防止资源在保存后被删除或禁用
2. 防止 supervisor 的 subagent 状态变化

策略：

1. 对正式聊天：返回明确错误
2. 对测试模式：错误直接在 SSE 中展示给页面

## 13. 风险点与处理建议

### 13.1 supervisor 循环引用

风险：

1. A -> B -> A
2. 或更长链路环

处理：

1. 保存和启用时都做 DAG 校验
2. 运行时再做一次兜底校验

### 13.2 测试污染正式数据

风险：

1. 创建页测试把消息写进正式 thread/message
2. 触发标题生成和长期记忆同步

处理：

1. debug SSE 走独立分支
2. `persist=false`
3. `skip_memory_sync=true`

### 13.3 agent 级长期记忆与现有 thread memory 语义冲突

风险：

1. 现有 memory 是 thread 导向
2. 新需求是 agent 导向

处理：

1. 把 memory 接口抽象为 scope
2. 不要继续在上层硬编码 thread-only

### 13.4 supervisor 绑定过多 subagent

风险：

1. prompt 噪声变大
2. tool 数量暴涨
3. 决策变慢

处理：

1. MVP 限制 subagent 数量，例如最多 8 个
2. 后续再考虑 supervisor 级 tool search

## 14. 分阶段实施建议

### Phase 1：后端 agent 配置实体 + 调试运行

目标：

1. 完成表结构
2. 完成 CRUD
3. 完成 `single_agent` 运行
4. 完成 `/v1/agents/debug/sse`

先不做：

1. supervisor
2. 聊天页 agent mode

### Phase 2：市场页 + 编辑页内测试

目标：

1. 完成 `/agents`、`/agents/new`、`/agents/[id]`
2. 打通测试面板
3. 打通启用/禁用

### Phase 3：supervisor_agent

目标：

1. subagent 绑定
2. 环检测
3. `AgentTool` 化运行

### Phase 4：聊天主页面接入 agent mode

目标：

1. 聊天页选择 enabled agent
2. 复用 `/v1/chat/sse`
3. 使用 agent 配置完成正式对话

## 15. 验收标准（DoD）

### 15.1 single_agent

1. 能创建、保存、启用、编辑、禁用
2. 能绑定模型、工具、skill、知识库
3. 能在创建/编辑页直接测试
4. 系统提示词在运行时真实生效

### 15.2 supervisor_agent

1. 能绑定 subagent
2. subagent 只能从 enabled agent 中选择
3. 运行时主 agent 能调用 subagent
4. 无环校验有效

### 15.3 市场页

1. 列表可看见全部 agent
2. 可搜索、筛选、启用、编辑、测试
3. 测试不污染正式历史

## 16. 结论

这个需求本质上不是“再做一个聊天页”，而是：

1. 把当前通用聊天底座升级成“可由 agent 定义驱动的聊天底座”
2. 在此之上补出 Agent 市场、编辑器和测试面板

结合当前仓库现状，最合适的技术路线是：

1. 继续使用 Eino `ChatModelAgent`
2. single agent 做成“配置化单 agent”
3. supervisor 做成“主 agent + AgentTool 化 subagent”
4. 复用现有模型、工具、skill、知识库、memory、SSE 能力
5. 用独立 debug SSE 解决“创建/编辑页内测试”的核心需求

如果你 review 这份方案没有问题，下一步建议先落 `Phase 1 + Phase 2`，也就是：

1. 先把 agent 实体与测试链路打通
2. 再做市场页
3. 最后再打开主聊天页里的 agent mode
