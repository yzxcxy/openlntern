# 上下文压缩功能调研与开发方案（openIntern）

## 1. 背景
当前聊天链路在服务端会将线程历史直接拼接进本轮输入：
- `openIntern_backend/internal/services/agent/agent_entry.go`
  - `ListThreadMessages(threadID)` 拉取全量历史
  - `mergeRunAgentInputHistory(...)` 合并历史 + 本轮用户消息
  - `applyForwardedPropsChain(...)` 注入 `contextSelections` 等额外上下文
  - `AGUIRunInputToEinoMessages(...)` 转为模型输入

当前实现没有统一的上下文预算（token budget）与压缩策略，存在以下问题：
- 会话越长，输入体积持续膨胀，模型响应延迟和成本上升。
- 没有“硬上限”保护，极长会话有触发模型上下文超限的风险。
- 工具检索、知识库注入、历史消息都在累加，缺少统一裁剪顺序。
- 历史中关键约束（用户偏好、已确认决策）没有结构化沉淀机制。

## 2. 在线调研结论（截至 2026-03-09）

### 2.1 主流做法已形成“组合拳”，不是单一算法
行业里基本都是混合策略：
- 滑动窗口（保留最近 N 轮）
- 历史摘要（rolling summary / session summary）
- 检索式回补（RAG memory）
- 预算感知裁剪（按 token 阈值触发）
- 结果缓存/提示缓存（降低重复上下文成本）

原因：单策略要么丢失关键信息，要么成本过高，混合策略在质量/成本/延迟之间更稳。

### 2.2 关键证据与可借鉴点
1. 长上下文并不等于有效利用
- “Lost in the Middle”显示模型对长上下文中间信息利用较弱，信息放在中间位置更容易被忽略。
- 启示：仅“加长上下文”不够，必须做结构化重排与压缩。

2. Prompt Compression（可选增强）
- LLMLingua 系列论文证明对输入进行有选择压缩可显著降低 token，同时保持任务效果。
- 启示：在工程上可以先做“语义级压缩”（摘要+裁剪），后续再考虑“token 级压缩器”作为可选插件。

3. 框架实践已把“上下文压缩”变成标准能力
- LangChain 的 `ContextualCompressionRetriever`（检索后压缩）是典型范式。
- LlamaIndex 的 Memory 模块提供 token limit、短期/长期块、自动 flush 机制。
- 启示：我们应把压缩抽象为独立模块，而不是散落在 controller/service 的 if-else 中。

4. Agent Runtime 开始内建会话压缩
- OpenAI Agents SDK Session 文档明确提供“自动修剪 + compact 触发”策略。
- Anthropic 文档提供 context editing（自动清理工具结果块）与 prompt caching（降低重复上下文成本）。
- 启示：服务端应支持自动压缩触发点，而不是仅依赖前端控制历史长度。

### 2.3 结论
对 openIntern 最适合的路线是：
- 第一阶段：预算感知的滑窗 + 硬上限防护（快速落地）
- 第二阶段：会话摘要（结构化）+ 摘要回填（质量稳定）
- 第三阶段：结合知识库/记忆检索做“按需回补”（减少误删关键信息）

## 3. 与本项目架构的匹配分析

### 3.1 最佳接入点
推荐把压缩放在 `applyForwardedPropsChain(...)` 之后、`AGUIRunInputToEinoMessages(...)` 之前：
- 此时历史与本轮输入都已合并。
- `contextSelections` 注入的 KB/Skill 信息已经就位，可被统一纳入预算。
- 可在同一处做“预算评估 -> 压缩 -> 输出”。

### 3.2 为什么不放前端
前端目前每轮只发送当前用户消息（历史由后端拼接），压缩若放前端会导致：
- 规则分散，后端仍无法保证硬上限。
- 不同客户端行为不一致。
- 无法统一复用模型信息（如 provider/model 对应预算）。

因此压缩应以后端为主，前端只做可视化提示（如“已压缩历史上下文”）。

### 3.3 与现有配置的关系
- `summary_llm` 已存在（目前用于标题生成），可复用于“摘要生成模型”。
- `models.ModelCatalog.CapabilitiesJSON` 可扩展 `context_window` 等字段，驱动预算。
- OpenViking 文档中已有 `SessionCompressor` 与 L0/L1/L2 思路，后续可作为增强方向。

## 4. 目标方案（不写代码版）

### 4.1 方案概览
新增一个“上下文编排器（Context Assembler）”，职责是：
1. 统一收集候选上下文（历史消息 + 注入上下文 + 当前用户消息）
2. 估算 token 占用并计算预算
3. 按策略压缩并返回最终消息序列

建议输出结构（逻辑层）：
- `Pinned`：不可删（system 指令、最后一条 user、必要工具状态）
- `RecentWindow`：最近 K 轮原文
- `SummaryBlock`：更早历史的结构化摘要
- `RetrievedMemory`：按需回补的长期记忆（可选）

### 4.2 压缩触发规则
- 先估算输入 token（按模型能力字段优先，缺失则用保守估算）
- 设置三段阈值：
  - `soft_limit`：超过后尝试压缩
  - `hard_limit`：超过后必须压缩
  - `safety_margin`：为输出 token 预留空间
- 若压缩后仍超限，执行降级：
  - 减小 recent window
  - 裁剪低优先级注入上下文
  - 最后返回明确错误（避免 silent failure）

### 4.3 摘要内容格式（建议）
摘要建议用稳定结构，避免自由文本漂移：
- `UserIntent`：用户目标
- `Decisions`：已确认决策
- `Constraints`：硬约束（技术/业务）
- `OpenTasks`：待办
- `Facts`：关键事实
- `DoNotRepeat`：已失败尝试

并以单条 `system` 或 `assistant` 记忆消息注入，前缀明确标记为“压缩摘要”。

### 4.4 存储建议
新增线程级压缩快照（建议单独表，不污染 `message`）：
- `thread_id`
- `compression_index`
- `covered_until_msg_id`（摘要覆盖到哪条消息）
- `summary_text`
- `summary_struct_json`
- `approx_tokens`
- `created_at`

作用：
- 避免每次都重新总结全部旧历史
- 支持增量压缩（只压缩新增区间）
- 可用于观测与回溯

### 4.5 多模态与工具消息处理
结合当前存储特点（历史合并主要吃 `text` 类型）：
- 工具调用/工具结果保留最近窗口原文，不做长期全量回放。
- 多模态历史（binary/url）在摘要中转成“文本事实”保留关键结论。
- Activity 消息默认不进入长期摘要，除非显式标记为关键结果。

## 5. 分阶段实施计划

### Phase 1：预算感知滑窗（1-2 天）
目标：先把“超限风险”降为可控。
- 新增上下文预算配置与估算器。
- 在后端组装阶段加入 `hard cap`。
- 保留最近窗口 + 最后一条用户消息 + 系统消息。
- 输出日志指标（压缩前后 token、裁剪数量、触发原因）。

交付结果：
- 不依赖新表，不依赖额外 LLM 调用，低风险上线。

### Phase 2：结构化摘要（2-4 天）
目标：在成本可控下提升长会话连续性。
- 引入摘要生成流程（优先复用 `summary_llm`）。
- 增量维护“线程压缩快照”。
- 组装上下文时插入 `SummaryBlock`。

交付结果：
- 中长会话质量明显稳定，token 增长斜率下降。

### Phase 3：检索式长期记忆回补（3-5 天）
目标：进一步减少“压缩误伤”。
- 从摘要快照/知识库中检索与当前 query 最相关的记忆片段回补。
- 与 `contextSelections` 统一排序与预算分配。

交付结果：
- 面对超长线程仍能保留关键历史事实。

## 7. 验收指标与观测
上线后建议重点观察：
- `input_tokens_before` / `input_tokens_after`
- `compression_trigger_rate`
- `summary_hit_rate`
- 首 token 延迟、整轮时延
- 模型报错率（context length exceeded）
- 人工抽检质量（是否丢失关键约束）

验收标准建议：
- 超长线程 context overflow 错误显著下降（目标接近 0）。
- 平均输入 token 降低（建议目标 30%+，按业务实测调整）。
- 核心任务成功率无明显下降。

## 8. 风险与对策
- 摘要失真
  - 对策：结构化 schema + 关键字段白名单 + 抽检回放。
- 压缩引入额外时延
  - 对策：Phase 1 先无 LLM 压缩；Phase 2 做增量摘要，避免每轮重算。
- 多模型预算不一致
  - 对策：`capabilities_json` 配置 `context_window`，无配置时走保守默认值。
