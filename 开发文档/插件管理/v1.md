# openIntern 插件管理开发执行文档

查阅日期：2026-03-01

本文档是插件管理能力的唯一方案文档。目标是先统一产品定义、协议边界和后端落地路径，确保前后端可以按同一套约束进入实现。

当前阶段只输出可执行方案，不直接落代码。

## 1. 目标

在现有 openIntern 能力上，补齐“插件管理”体系，形成一条闭环链路：

1. 前端提供统一的“插件广场”管理入口。
2. 插件按来源区分为“内建插件”和“自定义插件”。
3. 插件按运行方式区分为 `code`、`mcp`、`api`。
4. 聊天页在 `chat` 模式下支持插件选择。
5. 模型广场支持配置“默认工具集”。
6. 前端通过 AG-UI 的 `forwardedProps.agentConfig` 传递运行时选择结果。
7. 后端按请求解析插件选择结果，组装本次实际可用工具，再交给 Eino runner。

一句话定义：插件管理是“面向用户和配置的组织层”，真正参与运行的是工具集合。

## 2. 当前现状与核心缺口

当前项目已经具备以下基础：

1. 前端聊天页已通过 AG-UI 发起请求，并支持通过 `forwardedProps` 透传运行时配置。
2. 后端 `RunAgent` 已有 `applyForwardedPropsChain`，当前已解析 `a2uiAction` 和 `agentConfig` 中的部分字段。
3. 后端已存在 Eino runner 装配链路。
4. 系统已存在 `Skill` 相关前后端页面、接口与运行时能力。

当前缺口主要有四类：

1. 没有“插件”这一层的统一建模，现有只有 `Skill` 能力和工具装配能力，概念是分裂的。
2. 没有“模型默认工具集”这类模型到工具的显式绑定关系。
3. 没有“按请求装配插件工具”的运行时流程，现有 runner 仍以全局工具集为主。
4. 没有针对 `mcp`、`api`、`code` 三类插件的统一数据结构、同步机制和安全边界。

因此，本方案的核心不是“新增一个页面”，而是把“插件管理 -> 协议透传 -> 请求级工具装配”完整串起来。

## 3. 范围与非目标

### 3.1 本期范围

本期方案覆盖：

1. 插件、工具、Skill 的统一术语与边界。
2. 前端入口与交互方案。
3. `forwardedProps.agentConfig` 的协议扩展方案。
4. 后端请求级工具装配方案。
5. 数据模型与接口草案。
6. 安全、审计、运维要求。
7. 分阶段实施顺序与 MVP 验收标准。

### 3.2 本期不做

本期明确不做：

1. 不实现 `search` 模式的真实搜索能力。
2. 不引入完整的第三方插件商店同步机制。
3. 不展开插件商业化、审核、计费规则。
4. 不在第一版支持任意代码直接在主服务进程执行。
5. Skill和plugin以及tool不是一个东西，任何的修改不应该去影响Skill


## 4. 统一术语与关键决策

这是整份方案的基础。如果这里不统一，后续的数据结构和接口会持续打架。

### 4.1 术语定义

1. `tool`
   Agent 最终可调用的最小执行单元。真正进入 runner 的是 `tool`，不是 `plugin`。
2. `plugin`
   面向产品展示、权限控制和用户选择的组织抽象。一个 `plugin` 可以聚合一个或多个 `tool`。
3. `builtin`
   平台内建来源，表示该插件或工具由系统预置并由平台维护。
4. `custom`
   业务自定义来源，表示该插件由业务侧创建并受平台约束执行。
5. `runtimeType`
   插件运行方式，取值为 `code`、`mcp`、`api`。
6. `defaultToolSet`
   某个模型在 `select` 模式下的默认工具集合，本质是“模型到工具”的绑定关系。
7. `toolResponseMode`
   工具返回方式，取值为 `streaming` 或 `non_streaming`。其中当前 `code` 类型插件仅允许 `non_streaming`。
8. `pluginMode`
   聊天页插件选择模式，取值为 `select` 或 `search`。

### 4.2 必须固定的决策

1. `plugin` 不是运行时执行单元，`tool` 才是。
2. 用户可以“选择插件”，但后端必须在 runner 装配前把插件展开成 `tool` 列表。
3. “内建/自定义”是来源维度，不是运行方式。
4. “Code/MCP/API”是运行方式维度，不是来源。
5. 默认工具集不是独立运行模式，而是 `select` 模式下的默认初始工具集合。
6. 内建工具必须保持只读，不允许通过插件管理流程增删改。
7. 所有动态工具在进入 agent 前都必须适配成 Eino 可识别的工具接口。


## 5. 端到端设计

本节描述从“用户选择”到“后端执行”的完整链路，是本方案的主干。

### 5.1 前端产品入口

前端需要形成三个入口，各自职责必须分清：

1. 插件广场
   负责插件定义管理、来源展示、状态控制和详情查看。
2. 聊天页
   负责本次会话的插件选择，只表达“本次启用哪些插件/工具”。

#### 5.1.1 前端新增插件引导

前端的“新增插件”不能直接给用户一个大表单，而应设计为分步式创建向导。目标是先帮助用户选对插件类型，再约束用户输入一份可以直接落库并可被后端装配的合法配置。

推荐交互流程如下：

1. 入口
   在插件广场右上角提供主按钮 `新增插件`；若当前没有任何自定义插件，则在空状态中同时提供 `新建 API 插件`、`新建 MCP 插件`、`新建 Code 插件` 三个快捷入口。
2. 第一步：选择来源与运行方式
   用户先明确这是“新增自定义插件”，然后选择 `API`、`MCP`、`Code` 三种运行方式之一。
   `builtin` 插件必须保持只读，只能查看，不进入创建流程。
3. 第二步：填写基础信息
   所有类型共用字段：`插件名称`、`描述`、`图标`、`是否启用`。
4. 第三步：填写运行配置
   按 `runtimeType` 动态切换表单，只展示当前类型必需字段，不混放其他类型字段。
5. 第四步：预览并确认
   提交前展示将生成的 `plugin` 和 `tool` 关键信息，包括工具名、运行方式、响应模式、输入参数结构预览。

前端创建向导需要遵守以下原则：

1. 不让用户直接编辑 Eino 的 `ParamsOneOf`。
2. 不让用户直接手写完整 JSON Schema。
3. 用户填写的是平台自定义字段定义，后端再将其转换为 JSON Schema 2020-12。
4. 表单必须做即时校验，避免用户提交后才发现配置非法。

#### 5.1.2 按类型的前端表单约束

`API`、`MCP`、`Code` 三类插件的创建表单必须分开设计。

1. `API`
   这是引导最强的一类，必须使用向导式配置，而不是自由表单。
2. `MCP`
   重点是连接信息和同步状态，不要求用户手工维护工具 schema。
3. `Code`
   重点是语言选择、输入参数定义和代码内容，并明确其受沙箱约束且仅支持 `non_streaming`。

对于 `API` 插件，前端必须拆成以下配置步骤：

1. 先选择 `APIRequestType`
   当前仅允许 `GET` 和 `POST`。
2. 再选择 `ToolResponseMode`
   当前仅允许 `streaming` 和 `non_streaming`。
   这是创建时的必填项，因为后端需要据此选择 `APIInvokableTool` 或 `APIStreamableTool`。
3. 再填写请求目标
   包括 `RequestURL`、超时、鉴权配置引用（如果前端需要展示则只展示引用，不展示敏感明文）。
4. 最后填写参数定义
   用户通过字段配置器维护 `query`、`header`、`body` 的字段，而不是直接填写 JSON。

`API` 插件的前端表单必须根据 `APIRequestType` 做动态约束：

1. 若选择 `GET`
   仅展示 `query` 和 `header` 配置区。
   `body` 配置区必须隐藏或禁用，并在 UI 上明确提示“GET 类型不支持 body”。
2. 若选择 `POST`
   展示 `query`、`header`、`body` 三个配置区。
   `body` 可继续按平台支持的编码方式扩展，例如 JSON 或表单。

`API` 插件的前端表单必须根据 `ToolResponseMode` 做动态提示：

1. 若选择 `non_streaming`
   提示该工具将走同步执行器，一次性返回完整结果。
2. 若选择 `streaming`
   提示该工具将走流式执行器，要求上游 API 必须支持 SSE、分块传输或 NDJSON 这类可增量读取协议。
   如果后端后续提供协议类型字段，前端应在此步继续要求用户补充流协议类型。

参数定义区建议使用“字段配置器”，每个字段至少支持：

1. 字段名
2. 字段类型
3. 是否必填
4. 描述
5. 枚举值（可选）
6. 对象/数组的嵌套定义

前端还应提供两个辅助能力：

1. 实时预览
   在侧边栏展示最终将生成的入参结构预览，帮助用户理解模型真正可传入的参数。
2. 即时校验
   至少校验：名称是否为空、URL 是否合法、字段名是否重复、`GET` 是否错误配置了 `body`、`streaming` 是否缺少流式能力说明。

### 5.2 聊天页交互约束

聊天页只在 `conversation.mode = chat` 时展示插件选择栏。

插件选择栏只支持两种模式：

1. `select`
   用户显式选择本次要启用的插件和工具。
2. `search`
   本期仅做 UI 占位，用于未来扩展；当前不触发真实搜索逻辑。

`select` 模式的行为必须明确：

1. 页面进入时，先读取当前的默认工具集，作为初始选中结果。
2. 用户可以在初始值基础上增减插件和工具。
3. 用户的显式选择优先级高于默认工具集。
4. 聊天请求发送时，只上传“选择结果”，不上传敏感配置。

建议前端状态至少包含：

1. `pluginMode`
2. `selectedPluginIds`
3. `selectedToolIds`
4. `defaultToolIds`
5. `availablePlugins`
6. `availableTools`
7. `pluginLoading`
8. `pluginError`

### 5.3 AG-UI 协议约定

运行时配置统一收敛在 `forwardedProps.agentConfig` 下，不新增新的顶层散键。

推荐结构如下：

```json
{
  "agentConfig": {
    "conversation": {
      "mode": "chat"
    },
    "model": {
      "providerId": "provider-id",
      "modelId": "model-id"
    },
    "plugins": {
      "mode": "select",
      "selectedPluginIds": [],
      "selectedToolIds": []
    },
    "features": {}
  }
}
```

字段约束：

1. `agentConfig.conversation.mode`
   继续沿用现有约定，控制会话模式。
2. `agentConfig.model`
   继续沿用现有约定，指定模型路由信息。
3. `agentConfig.plugins.mode`
   支持 `select` 和 `search`。其中 `search` 当前只作为占位标志，不进入真实工具装配。
4. `selectedPluginIds`
   仅上传插件 ID。
5. `selectedToolIds`
   仅上传工具 ID，用于表达最终启用的工具集合或对已选插件做工具级过滤。
6. `features`
   保持为预留扩展字段，不和插件选择混用。

协议边界：

1. 前端可以传“选择结果”，但不能传插件凭据、API Key、Base URL、MCP 凭据等敏感信息。
2. 前端不能通过 `forwardedProps` 新增工具定义，也不能修改内建工具定义。

### 5.4 后端运行时链路

后端必须采用“按请求装配工具”的方案，不能通过修改全局工具列表来响应用户选择。

推荐链路如下：

1. `RunAgent` 收到请求后先完成现有的历史消息合并。
2. `applyForwardedPropsChain` 解析 `agentConfig`，在现有 `conversation`、`model`、`features` 基础上补充 `plugins`。
3. 若 `plugins.mode` 为空，视为本次未启用插件选择，则不给agent加入任何工具。
4. 若 `plugins.mode = search`，仅记录占位状态，本期不进入真实插件装配。
5. 若 `plugins.mode = select`，则进入请求级工具装配流程。
8. 根据传入 `selectedToolIds`，在工具层执行最终过滤；显式工具选择覆盖默认初始值。
9. 对过滤后的工具逐个校验 schema、权限、状态和返回模式。
   若 `runtimeType = code`，还必须强制校验 `toolResponseMode = non_streaming`。
10. 将最终工具定义适配为 Eino 支持的工具接口。
11. 与系统基础工具合并，生成本次请求专用的 runner。

关键原则：
1. 插件选择的全部影响范围都必须被限制在当前请求上下文内。

### 5.5 三类插件的运行方式

#### 5.5.1 `code`

`code` 类型插件的核心约束是“受控执行”。

推荐方式：

1. 插件在装配阶段被转换为受控工具代理，并明确脚本语言为 `python` 或 `javascript`，这个脚本语言类型在tool的数据库中存储。
2. 工具真正执行前，后端从服务端配置文件中读取沙箱地址或基础路径，不允许前端透传或在插件配置中覆盖该地址。
3. 后端基于读取到的沙箱配置，调用沙箱对外暴露的代码执行接口，执行对应的 Python 或 JavaScript 代码，而不是在主服务内直接拉起解释器。
4. `code` 类型插件当前只支持同步结果回传，不支持流式传输，因此其 `toolResponseMode` 必须固定为 `non_streaming`。
5. 沙箱负责隔离、资源限制、超时和结果回传。
6. 主服务进程不直接执行插件代码。

参考接口：

1. `https://sandbox.agent-infra.com/zh/api#tag/code`

#### 5.5.2 `mcp`

`mcp` 类型插件的核心是异步同步schema和是否有新的tool,这是为了能够在前端及时展示最新的工具定义。

对于mcp定义的同步是异步执行的，这需要其一个启动一个协程，从延时队列中拿到同步任务，然后执行同步操作。

同时在createPlugin操作中，如果是mcp类型，也会将该任务加入延时队列中，同时还会启动一个定时器，定时加入一批mcp plugin到延时队列中。

延时队列可以使用redis的zset来进行实现。

同步内容至少包含工具名、描述、输入 schema、输出 schema、状态和同步时间。

而对于mcp插件的过滤和执行，首先，拿到mcp地址，然后使用eino中的方法去过滤出需要的mcp工具，然后加入到此次执行的工具集合中，具体见：https://cloudwego.io/zh/docs/eino/ecosystem_integration/tool/tool_mcp/

#### 5.5.3 `api`

`api` 类型插件的核心约束是“受控 HTTP 工具”，不是前端自由拼接请求。

同时，`api` 工具不能直接套用 `utils.InferTool` 这类静态结构体推导方案，因为其入参 schema 来自数据库配置，必须在运行时动态生成；因此应手动实现 Eino 的工具接口：

1. `BaseTool.Info(ctx)` 负责返回动态生成的 `schema.ToolInfo`。
2. `non_streaming` 模式实现 `InvokableTool.InvokableRun(...)`。
3. `streaming` 模式实现 `StreamableTool.StreamableRun(...)`。

推荐方式：

1. 服务端保存 API 地址、请求类型、响应模式、超时、鉴权配置引用以及参数模板；其中请求类型当前只支持 `GET` 和 `POST`。
2. `api` 工具在创建时必须显式选择 `toolResponseMode = streaming` 或 `non_streaming`，后端装配时据此选择对应执行器，不能依赖隐式推断。
3. API 插件的入参统一拆分为 `query`、`header`、`body` 三段，由平台自定义 schema 描述，而不是让前端或模型直接拼任意 JSON。
4. 工具装配时按定义生成受控 API 调用器。
5. 聊天请求阶段只传插件或工具选择，不传 API 目标地址和敏感头信息。

##### API 插件的入参 schema 约定

1. 平台内部需要为 `api` 工具定义一套独立 schema 模型，至少覆盖：字段名、字段位置（`query`/`header`/`body`）、字段类型、是否必填、描述、枚举值，以及对象/数组的嵌套定义。
2. 数据库存储建议按三段分开保存：`QuerySchemaJSON`、`HeaderSchemaJSON`、`BodySchemaJSON`；每段都是平台自定义的字段定义，不直接暴露为最终给模型的 JSON Schema。
3. `Info(ctx)` 返回给 Eino 时，需要把这三段平台自定义定义转换为一个统一的根对象 schema，再通过 `schema.NewParamsOneOfByJSONSchema(...)` 填入 `ToolInfo.ParamsOneOf`。
4. 根对象固定只允许出现 `query`、`header`、`body` 三个一级字段；未启用的段不生成到 `properties` 中。
5. 每个段内部再展开为对应的 JSON Schema 2020-12 `object`，字段 required 规则按平台配置映射。
6. 根对象以及各段对象都应显式设置 `additionalProperties = false`，防止模型生成未定义参数。
7. `header` 段只用于暴露允许模型填写的业务头；认证头、签名头、固定系统头一律由服务端在执行阶段注入，不能放进可调用 schema。
8. 若工具的请求类型是 `GET`，则只允许定义 `query` 和 `header`，`body` 必须为空且不能出现在最终 schema 中。
9. 若工具的请求类型是 `POST`，则允许定义 `body`，同时 `query`、`header` 仍可按需存在。

推荐转换后的入参形态如下：

```json
{
  "type": "object",
  "additionalProperties": false,
  "properties": {
    "query": {
      "type": "object",
      "additionalProperties": false,
      "properties": {
        "keyword": {
          "type": "string",
          "description": "搜索关键词"
        },
        "page": {
          "type": "integer",
          "description": "页码"
        }
      },
      "required": ["keyword"]
    },
    "header": {
      "type": "object",
      "additionalProperties": false,
      "properties": {
        "x_trace_id": {
          "type": "string",
          "description": "业务透传链路 ID"
        }
      }
    },
    "body": {
      "type": "object",
      "additionalProperties": false,
      "properties": {
        "user_id": {
          "type": "string",
          "description": "用户 ID"
        }
      },
      "required": ["user_id"]
    }
  }
}
```

##### `Info(ctx)` 的职责

1. 从工具定义中读取 `ToolName`、`Description`、`QuerySchemaJSON`、`HeaderSchemaJSON`、`BodySchemaJSON`。
2. 将平台自定义的三段 schema 递归转换为 JSON Schema 2020-12；如果字段类型超出平台支持范围，应直接返回错误，阻止该工具进入 runner。
3. 组装 `schema.ToolInfo{Name, Desc, ParamsOneOf: schema.NewParamsOneOfByJSONSchema(...)}`。
4. `api` 工具的 `Info(ctx)` 必须保证每次都能从当前数据库定义重建 schema，避免缓存陈旧定义导致模型看到的参数和真实执行参数不一致。

##### 非流式执行器（`InvokableTool`）

1. `toolResponseMode = non_streaming` 时，`api` 工具应实现 `InvokableRun(ctx, argumentsInJSON string, opts ...Option) (string, error)`。
2. `InvokableRun` 首先把 `argumentsInJSON` 解析为统一输入结构，例如：

```go
type APIInvokeInput struct {
    Query  map[string]any `json:"query,omitempty"`
    Header map[string]any `json:"header,omitempty"`
    Body   any            `json:"body,omitempty"`
}
```

3. 解析后必须按 `Info(ctx)` 生成时的同一份 schema 再做一次服务端校验，避免模型绕过约束传入非法字段或错误类型。
4. 执行时必须按工具记录的请求类型分支处理：`GET` 只将 `query` 编码到 URL querystring 并忽略/拒绝 `body`；`POST` 则将允许透传的 `header` 合并到请求头，并把 `body` 按工具定义编码为 JSON、表单或原始文本。
5. 服务端再补齐固定认证头、签名头、超时控制、重试策略和审计字段，这些都不从模型输入读取。
6. 发起受控 HTTP 请求，拿到响应后按工具定义做状态码校验和响应体截断，避免超大响应直接返回给模型。
7. 返回值仍是字符串：如果上游返回 JSON，优先返回原始 JSON 字符串；如果需要保留元信息，则统一封装为字符串化的 JSON，例如 `{"statusCode":200,"body":...}`。

##### 流式执行器（`StreamableTool`）

1. `toolResponseMode = streaming` 时，`api` 工具应实现 `StreamableRun(ctx, argumentsInJSON string, opts ...Option) (*schema.StreamReader[string], error)`。
2. 入参解析、schema 校验、请求拼装与非流式执行器完全一致，不能维护两套不同的参数解释逻辑。
3. 只有当上游 API 明确支持 SSE、分块传输或 NDJSON 这类可增量读取协议时，才允许将工具标记为 `streaming`；否则在装配阶段直接判定为非法配置。
4. `StreamableRun` 在成功建立上游连接后，通过 `schema.Pipe[string]` 创建流，并启动 goroutine 持续读取上游响应流。
5. 读取策略应按协议做适配：SSE 读取 `data:` 事件，NDJSON 按行读取，普通 chunked 则按块读取；每个可消费片段都转换为字符串后写入 `StreamWriter`。
6. 上游结束时关闭 `StreamWriter`；若中途出现网络错误、超时或解析错误，应将错误写入流并终止，确保 Eino 可以感知流结束。
7. 如果上游虽然支持流式，但最终返回的是一个完整 JSON 块，也可以按单块发送一次后关闭流，保持接口一致性。

##### API 执行器示意

推荐拆成两个具体执行器，而不是在一个结构体里混合判分支：

```go
type APIInvokableTool struct {
    def    APIToolDefinition
    client *http.Client
}

type APIStreamableTool struct {
    def    APIToolDefinition
    client *http.Client
}
```

两者共享同一套 schema 转换、参数解析和请求构造逻辑，只在“读取上游响应并回传给 Eino”的阶段分叉。

#### 5.5.4 统一分配器与各类型执行器

这样子在装配为Eino插件的时候，只需要根据插件的类型，填入对应的配置，加上对应的执行器即可。

推荐按两层路由来做：

1. 第一层按 `runtimeType` 选择 `code`、`mcp`、`api` 对应的执行器工厂。
2. 第二层仅在 `api` 类型内按 `APIRequestType + toolResponseMode` 进一步选择具体执行器；其中 `toolResponseMode = non_streaming` 选择 `APIInvokableTool`，`toolResponseMode = streaming` 选择 `APIStreamableTool`。
3. 工厂统一返回 `einoTool.BaseTool`，但实际对象必须额外满足对应的 `InvokableTool` 或 `StreamableTool` 接口。
4. 这样 runner 装配层只关心“拿到一个合法的 Eino 工具”，不用关心插件来源和具体执行细节。

### 5.6 工具级通用要求

无论工具来自 `builtin`、`skill`、`mcp`、`api` 还是 `code`，都必须满足：

1. 有稳定的 `ToolID`。
2. 有明确的输入 schema。
3. 有明确的输出 schema 或输出结构说明。
4. 有合法的 `toolResponseMode`。`mcp`、`api` 工具可按能力声明 `streaming` 或 `non_streaming`；当前 `code` 工具只能声明 `non_streaming`。
5. 能被转换为 Eino 可识别的工具接口。

## 6. 数据模型与接口草案

本节只定义 MVP 需要的最小数据结构，目标是支撑管理、展示和请求级装配。

### 6.1 推荐数据模型

1. `t_plugin`
   保存插件定义。建议字段包括：`ID`、`PluginID`、`Name`、`Description`、`Icon`、`Source`、`RuntimeType`、`Status`、`MCPUrl`、`MCPProtocol`、`CreatedAt`、`UpdatedAt`。
2. `t_tool`
   保存插件展开后的工具定义，是运行时装配的直接来源。建议字段包括：`ToolID`、`PluginID`、`ToolName`、`Description`、`InputSchemaJSON`、`OutputSchemaJSON`、`ToolResponseMode`、`Enabled`、`Code`、`CodeLanguage`、`CreatedAt`、`UpdatedAt`。
   若 `runtimeType = api`，还需要额外保存：`APIRequestType`、`RequestURL`、`QuerySchemaJSON`、`HeaderSchemaJSON`、`BodySchemaJSON`、`AuthConfigRef`、`TimeoutMS`。
   其中 `APIRequestType` 是必填枚举，当前仅允许 `GET` 和 `POST`，并直接决定该工具的参数结构与执行分支。
   对 `runtimeType = api` 而言，`ToolResponseMode` 同样是必填配置，前端创建时必须让用户显式选择 `streaming` 或 `non_streaming`，后端据此装配对应执行器。
4. `t_plugin_default`
   保存模型维度的默认工具集。建议字段包括： `ID`,`ToolID`。

必须遵守的拆分原则：

1. 插件定义和插件绑定分开存。
2. 插件定义和工具定义分开存。
3. 默认工具集单独存，不和插件定义混放。


### 6.3 推荐接口草案

插件管理接口：

1. `GET /v1/plugins`
   获取插件列表，支持按来源、类型、关键字、状态筛选。
2. `GET /v1/plugins/:pluginId`
   获取插件详情。
3. `POST /v1/plugins`
   创建自定义插件定义。
4. `PUT /v1/plugins/:pluginId`
   更新插件定义。
5. `POST /v1/plugins/:pluginId/enable`
   启用插件。
6. `POST /v1/plugins/:pluginId/disable`
   禁用插件。
7. `POST /v1/plugins/:pluginId/sync`
   手动触发一次 `mcp` 插件同步。
8. `DELETE /v1/plugins/:pluginId`
   删除插件定义。

运行时辅助接口：

1. `GET /v1/plugins/available-for-chat`
   返回当前用户在聊天页可选的插件及轻量工具信息。

## 7. 安全、审计与运维要求

插件体系如果没有服务端约束，会直接成为系统的安全缺口。这部分不能弱化。

### 7.1 前端传输边界

前端允许上传：

1. 插件 ID。
2. 工具 ID。
3. 选择模式。
4. 非敏感运行时开关。

前端禁止上传：

1. API Key。
2. Base URL。
3. 自定义 Header 明文。
4. MCP Server 凭据。
5. Code 插件源码或执行脚本内容。
6. 新工具定义或内建工具修改内容。

### 7.2 服务端必做控制点

服务端必须负责：

1. 插件凭据安全存储。
2. 插件存在性与可用性校验。
3. 默认工具集读取与校验。
4. 工具输入输出 schema 校验。
5. `toolResponseMode` 校验。
6. Eino 工具适配校验。
7. `mcp` 同步任务与同步结果有效性校验。
8. `code` 插件沙箱执行校验，包括沙箱配置读取、语言白名单和执行接口可用性校验。
10. 工具超时
11. 内建工具只读约束。


## 8. 分阶段实施与验收

本节定义实际开发顺序，确保可以从现状平滑推进，而不是一次性改穿整条链路。

### 8.1 实施原则

实施时必须遵守以下原则：

1. 优先按“数据结构 -> 管理接口 -> 前端管理页 -> 运行时装配 -> 类型能力补齐”的顺序推进，不要一开始就同时改穿整条链路。
2. 后端插件相关的大部分实现应放在 `internal/services/tools` 及其相关子目录中，避免把插件装配、执行器、schema 转换逻辑过度堆到 `agent` 主链路文件里。
3. `RunAgent` 只保留请求级装配入口和最薄的一层调用，不承担插件定义管理、API 请求拼装、MCP 同步等细节。
4. 每个阶段都必须能单独联调和验证，不能把验收全部推迟到最后。
5. 每个阶段结束时，都要保证现有非插件能力不回归，尤其是现有聊天链路和基础工具链路。

### 8.2 阶段一：数据模型与后端基础设施

本阶段目标是先把“插件”和“工具”的存储、查询、校验骨架搭起来，为后续前端和运行时提供稳定基础。

范围：

1. 落地 `t_plugin`、`t_tool`、`t_plugin_default` 的数据库结构。
2. 明确 `runtimeType`、`APIRequestType`、`ToolResponseMode` 的枚举约束。
3. 补齐插件管理的最小后端接口骨架：列表、详情、创建、更新、启停、删除。
4. 完成后端的插件定义校验逻辑，包括：
   `api` 类型的 `GET/POST` 校验、`streaming/non_streaming` 校验、`GET` 不允许 `body` 的校验。
5. 先不接入真实 runner，只保证“定义可存、可查、可改、可校验”。

验收标准：

1. 可以通过接口创建一个合法的 `api` 插件定义，并成功写入 `t_plugin` 与 `t_tool`。
2. `api` 工具创建时，若缺少 `APIRequestType` 或 `ToolResponseMode`，后端必须拒绝写入。
3. `api` 工具创建时，若配置为 `GET` 且存在 `BodySchemaJSON`，后端必须拒绝写入。
4. 内建插件不能通过创建、更新、删除接口被修改。
5. 不改动现有聊天运行链路时，系统原有能力保持可用。

### 8.3 阶段二：前端插件管理页与新增插件引导

本阶段目标是让用户能够在前端完成“创建、查看、编辑、启停”插件定义，并通过引导正确填写配置。

范围：

1. 在插件广场提供列表页、详情页、状态切换和删除入口。
2. 落地“新增插件”分步式创建向导。
3. 先支持 `custom + api` 的完整创建流程，再补 `mcp` 与 `code` 的基础表单。
4. 在 `api` 向导中强制用户显式选择：
   `APIRequestType = GET/POST` 与 `ToolResponseMode = streaming/non_streaming`。
5. 提供 `query/header/body` 字段配置器、实时预览和即时校验。

验收标准：

1. 用户可以在前端完整创建一个 `api` 插件，并成功提交到后端。
2. 当前端选择 `GET` 时，`body` 配置区必须隐藏或禁用。
3. 当前端未选择 `ToolResponseMode` 时，不允许提交。
4. 当字段名重复、URL 非法或必填项缺失时，表单能在提交前给出明确错误提示。
5. 插件列表页能正确展示来源、类型、状态和工具摘要信息。

### 8.4 阶段三：聊天页接入与请求级工具装配

本阶段目标是打通“前端选择 -> AG-UI 透传 -> 后端请求级装配 -> runner 执行”的主链路，但优先只覆盖 `api` 插件。

范围：

1. 在聊天页接入插件选择栏，并按 `select/search` 模式透传 `forwardedProps.agentConfig.plugins`。
2. 后端在 `applyForwardedPropsChain` 中补充对 `plugins` 的解析。
3. 按请求读取 `selectedPluginIds`、`selectedToolIds` 并展开为本次请求工具列表。
4. 完成 `api` 工具的 Eino 适配：
   动态 `Info(ctx)`、`ParamsOneOf` 转换、执行器装配。
5. 先打通 `api + non_streaming`，再接 `api + streaming`。

验收标准：

1. 聊天请求未选择插件时，不影响现有默认对话能力。
2. 聊天请求选择某个 `api` 工具后，后端仅在当前请求中装配该工具，不污染全局工具集。
3. `api + non_streaming` 工具能完成一次真实调用并返回结果给模型。
4. `api + streaming` 工具能在上游支持流式协议时，返回可消费的流给 Eino。
5. 若工具 schema 非法、插件已禁用或请求配置与工具定义不匹配，后端能在装配阶段直接拒绝。

### 8.5 阶段四：补齐 `mcp` 与 `code` 类型能力

本阶段目标是将插件体系从“仅 API 可运行”扩展到覆盖三类运行方式。

范围：

1. 落地 `mcp` 插件的同步任务模型、延时队列、定时补偿和手动同步。
2. 将 `mcp` 同步结果写入 `t_tool`，用于前端展示和运行时过滤。
3. 接入 `mcp` 工具过滤与运行时装配。
4. 落地 `code` 插件的创建表单、后端定义校验和沙箱执行代理。
5. 强制 `code` 工具只能使用 `non_streaming`。

验收标准：

1. 创建 `mcp` 插件后，系统能够触发首次同步并在前端看到同步出的工具列表。
2. 手动触发 `POST /v1/plugins/:pluginId/sync` 后，可以刷新工具定义和同步时间。
3. 选择 `mcp` 插件后，后端能够只装配用户选中的 MCP 工具。
4. `code` 插件可以通过沙箱完成一次受控执行，并返回结果。
5. 若 `code` 插件被配置为 `streaming`，后端必须拒绝。

### 8.6 阶段五：安全加固、审计补齐与上线验收

本阶段目标是把系统从“功能可用”推进到“可上线运维”。

范围：

1. 补齐插件创建、更新、启停、删除、同步、执行的审计日志。
2. 为 `api`、`mcp`、`code` 补齐超时、重试、熔断和错误归类。
3. 补齐敏感配置的加密存储与脱敏展示。
4. 补齐回归测试，覆盖管理接口、前端向导、运行时装配和执行器分支。
5. 输出上线清单和回滚方案。

验收标准：

1. 插件管理的关键操作均可追踪到审计记录。
2. 敏感字段不会出现在前端透传、接口响应明文或日志明文中。
3. 三类插件在禁用、删除、配置非法、上游失败时都能给出可预期错误。
4. 核心回归测试通过，且现有聊天主链路无明显行为回归。
5. 满足 MVP 上线条件后，方可进入正式灰度。

### 8.7 MVP 验收口径

若需要先交付一版最小可用能力，则 MVP 只要求完成到“阶段三”的 `api` 能力闭环，不要求 `mcp` 与 `code` 全量上线。

MVP 的最低验收标准如下：

1. 用户可以在前端创建一个 `custom api` 插件。
2. 创建时必须选择 `GET/POST` 与 `streaming/non_streaming`。
3. 聊天页可以选择该插件，并将选择结果通过 `forwardedProps.agentConfig.plugins` 透传给后端。
4. 后端可以按请求将该插件展开为 Eino 工具，并根据 `ToolResponseMode` 选择正确执行器。
5. 至少打通一条 `api + non_streaming` 的真实调用链路。
6. 若上游 API 支持流式协议，则 `api + streaming` 可用；若暂未接入真实流式上游，则至少完成执行器和联调桩验证。
