# v0.0.2 升级 Eino 版本为 0.8.x

查阅与实施日期：2026-03-21

本文档记录 openIntern 第一阶段的 Eino 升级实施内容。当前服务尚未上线，因此本次升级不以“最大兼容旧实现”为目标，而是优先把运行时结构切到 Eino 0.8.x 推荐的 ADK middleware/handler 体系上。

## 1. 升级目标

第一阶段只解决两类问题：

1. 将后端 `github.com/cloudwego/eino` 从 `v0.7.34` 升级到 `v0.8.4`。
2. 将 Agent 运行时从旧的 `[]adk.AgentMiddleware + Middlewares` 切换到 0.8.x 推荐的 `[]adk.ChatModelAgentMiddleware + Handlers`。

本阶段不处理：

1. 自定义上下文压缩逻辑的大重构。
2. 插件工具搜索逻辑向官方 `toolsearch` middleware 的彻底迁移。
3. AG-UI 事件桥接逻辑的重写。

## 2. 参考依据

本次升级主要参考以下资料：

1. Eino 官方仓库：[cloudwego/eino](https://github.com/cloudwego/eino)
2. Eino 0.8 不兼容更新说明：[官方迁移文档](https://www.cloudwego.io/zh/docs/eino/release_notes_and_migration/eino_v0.8._-adk_middlewares/eino_v0.8_%E4%B8%8D%E5%85%BC%E5%AE%B9%E6%9B%B4%E6%96%B0/)
3. 本仓库本地参考实现：
   - `_eino/eino/adk/chatmodel.go`
   - `_eino/eino/adk/middlewares/skill/skill.go`
   - `_eino/eino/adk/middlewares/patchtoolcalls/patchtoolcalls.go`
   - `_eino/eino-ext/skills/eino-agent/reference/middleware.md`

关键判断如下：

1. `v0.8.4` 是实际存在的可解析版本。
2. 0.8.x 之后，`ChatModelAgent` 的主扩展方式应优先使用 `Handlers []adk.ChatModelAgentMiddleware`。
3. `skill` middleware 在 0.8.x 应优先使用 `skill.NewMiddleware(...)`，旧的 `skill.New(...)` 仍保留，但属于兼容层接口。
4. `patchtoolcalls` 是 0.8.x 建议优先放在 middleware 链最前面的内建能力，适合当前项目这种“消息历史来自数据库回放”的场景。

## 3. 第一阶段实施方案

### 3.1 依赖版本调整

后端模块 `openIntern_backend/go.mod` 升级为以下组合：

1. `github.com/cloudwego/eino v0.8.4`
2. `github.com/cloudwego/eino-ext/components/model/ark v0.1.65`
3. `github.com/cloudwego/eino-ext/components/model/openai v0.1.10`
4. `github.com/cloudwego/eino-ext/components/model/deepseek v0.1.2`
5. `github.com/cloudwego/eino-ext/components/tool/mcp v0.0.8`
6. `github.com/cloudwego/eino-ext/callbacks/apmplus v0.0.2`
7. `github.com/cloudwego/eino-ext/libs/acl/openai v0.1.14`

说明：

1. `deepseek`、`mcp`、`apmplus` 在当前阶段已经是可用上限版本，因此不额外抬升。
2. `ark`、`openai`、`acl/openai` 随 core 一起提升到 0.8.x 时代常用组合，避免只升 core 导致 ext 侧接口滞后。
3. `go mod tidy` 后，间接依赖 `sonic`、`doublestar` 等也会被联动更新，这是正常结果。

### 3.2 Agent 运行时抽象切换

将当前项目的 agent 运行时状态从旧结构切换到新结构：

1. `runtimeState.agentMiddlewares []adk.AgentMiddleware`
2. 切换为 `runtimeState.agentHandlers []adk.ChatModelAgentMiddleware`

对应原因：

1. 0.8.x 的内建中间件如 `patchtoolcalls`、`toolsearch`、`summarization`、`skill` 的主接入形态都是 `ChatModelAgentMiddleware`。
2. 项目后续还要继续对齐 `toolsearch`、`summarization`，如果继续停留在 `AgentMiddleware`，后面还得再拆一次。

### 3.3 Skill middleware 升级

将技能中间件接入方式从：

1. `skill.New(ctx, &skill.Config{...})`

切换为：

1. `skill.NewMiddleware(ctx, &skill.Config{...})`

并将中文提示语的控制方式从单个 middleware 的 `UseChinese` 切换为：

1. `adk.SetLanguage(adk.LanguageChinese)`

这样做的原因：

1. 更符合 0.8.x 的官方写法。
2. 后续如果再引入 `summarization`、`patchtoolcalls`、`toolsearch` 等内建 middleware，它们也会统一遵循全局语言设置。

### 3.4 引入 patchtoolcalls middleware

在 agent 初始化时新增：

1. `patchtoolcalls.New(ctx, nil)`

并将其放在 handler 链最前面：

1. `patchToolCallsMiddleware`
2. `skillMiddleware`

这样做的原因：

1. 当前项目的对话上下文来自数据库消息回放与再拼接。
2. 在用户中断、工具调用未闭合、历史消息不完整时，最容易出现 dangling tool calls。
3. `patchtoolcalls` 应先于其他 middleware 运行，先把消息历史修补干净，再交给 skill 和后续 middleware 继续处理。

### 3.5 ChatModelAgent 配置字段切换

将 Agent 创建配置从：

1. `Middlewares: state.agentMiddlewares`

切换为：

1. `Handlers: state.agentHandlers`

这是本次第一阶段的核心接口切换点。只有先切到 `Handlers`，后续第二阶段接官方 `toolsearch`、第三阶段接官方 `summarization` 才不会再推倒重来。

## 4. 本阶段实际修改文件

本次第一阶段已实际修改以下文件：

1. `openIntern_backend/go.mod`
2. `openIntern_backend/go.sum`
3. `openIntern_backend/internal/services/agent/agent_service.go`
4. `openIntern_backend/internal/services/agent/agent_init.go`
5. `openIntern_backend/internal/services/agent/agent_runner.go`

各文件职责如下：

1. `go.mod / go.sum`
   负责依赖升级与锁定。
2. `agent_service.go`
   将运行时状态从旧的 middleware 类型切换到新的 handler 类型。

4. `agent_runner.go`
   负责在 `adk.NewChatModelAgent` 中使用 `Handlers` 字段接入新链路。

## 7. 第二阶段：自定义 tool_search 重构

实施日期：2026-03-22

第一阶段完成后，项目已经切到了 `Handlers` 体系。第二阶段不再继续保留旧的“后端预搜索并直接把最终工具注入模型”的实现，而是重构为“自定义 `tool_search` + 动态工具可见性控制”。

### 7.1 不采用官方 toolsearch middleware 的原因

这次没有直接接入官方 `adk/middlewares/dynamictool/toolsearch`，原因如下：

1. 官方实现更适合“在已有候选工具集合里按名字或正则过滤”。
2. openIntern 的插件搜索核心能力来自 OpenViking 语义召回，而不是工具名正则匹配。
3. 当前项目更需要的是“让模型在运行过程中主动发起一次语义工具检索”，而不是让后端在每轮开始前替模型把工具决定好。

因此，第二阶段采用的是自定义实现：

1. 自定义 `tool_search` 元工具负责调用 OpenViking 检索工具。
2. 自定义 `ChatModelAgentMiddleware` 负责把预装的动态工具按需暴露给模型。

### 7.2 新的运行时结构

运行时工具集合明确拆成两类：

1. 静态工具集合
   永远对模型可见，包括：
   - A2UI 工具
   - COS 工具
   - skill 文件工具
   - sandbox MCP 工具
   - 搜索模式下额外注入的 `tool_search`
2. 动态工具集合
   在每次 agent run 开始时就预装到执行层，但默认不对模型可见，包括：
   - 全部启用态 API 插件工具
   - 全部启用态 Code 插件工具
   - 全部启用态 MCP 插件工具

这里“预装”是为了对齐 Eino 0.8.x 的执行模型：

1. `BeforeAgent` 可以修改真实可执行工具集合。
2. `WrapModel` 只能修改发送给模型的工具信息列表，不能在运行中再创建新的真实工具实例。

因此，本阶段采用的是“预装执行层工具，再动态控制模型可见性”的方案，而不是“搜索后再创建工具实例”。

### 7.3 自定义 tool_search 参数面

暴露给 LLM 的参数只保留最小集合：

```json
{
  "intent": "查询武汉实时天气",
  "keywords": ["武汉", "天气"],
  "top_k": 4
}
```

参数含义：

1. `intent`
   必填，用于表达当前需要的工具能力。
2. `keywords`
   可选，用于补充实体词。
3. `top_k`
   可选，范围限制为 `1-8`，默认 `4`。

不再向 LLM 暴露以下检索控制参数：

1. `runtime_types`
2. `min_score`
3. `max_mcp_tools`
4. `target_uri`
5. 任何排序或阈值控制参数

这些都属于后端检索策略，不应由模型参与配置。

### 7.4 tool_search 返回结构

`tool_search` 不再返回长描述、完整 schema 或插件详情，只返回后续可见性控制所需的最小结果：

```json
{
  "selected_tool_names": ["get_weather", "get_forecast"]
}
```

这样做的原因：

1. 真正的工具描述会在下一轮通过 `WrapModel` 注入到模型可见工具列表中。
2. 如果 `tool_search` 自身再返回大段解释，会造成工具信息在上下文里重复出现。
3. 当前项目已经出现过“工具消息过大导致持久化失败”的问题，因此必须主动约束中间结果体积。

### 7.5 具体注入点选择

第二阶段最终确定的扫描与注入位置如下：

1. `BeforeAgent`
   将全部动态插件工具追加到真实执行工具集合。
2. `BeforeModelRewriteState`
   在 `search` 模式下扫描历史消息里的 `tool_search` tool result，解析出已经选中的动态工具名，并写入 `RunLocalValue`。`select` 模式只使用前端明确选中的工具，不读取历史 `tool_search` 结果。
3. `WrapModel`
   读取 `RunLocalValue`，只把：
   - 静态工具
   - 当前已选中的动态工具
   暴露给模型。

之所以把“扫描搜索结果”的逻辑放在 `BeforeModelRewriteState`，而不是 `WrapModel` 或其他位置，原因是：

1. 它天然能看到当前完整的 `state.Messages`。
2. 它发生在每次模型调用之前，正好可以响应上一个 tool call 的结果。
3. 扫描结果写入 `RunLocalValue` 后，`WrapModel` 就只需要做纯过滤，不需要重复解析消息。

### 7.6 插件层改造

为了支持新运行时结构，插件层同步做了两项调整：

1. 新增显式的全量 runtime builder
   - `BuildAllRuntimeCodeTools`
   - `BuildAllRuntimeAPITools`
   - `BuildAllRuntimeMCPTools`

   这样“按 `toolIDs` 构建”和“预装全部动态工具”是两套明确接口，不再通过“传空切片”这种隐式语义区分。
2. 新增启用态工具名映射能力
   允许在 `select` 模式下将 `selectedToolIDs` 解析为工具名，并作为初始可见动态工具集合。

另外，OpenViking 搜索逻辑仍然保留，但职责发生变化：

1. 旧实现：后端在 agent 启动前搜索工具，并把搜索结果直接注入模型。
2. 新实现：`tool_search` 在运行过程中调用 OpenViking 搜索，返回最小命中集合，再由 middleware 控制这些工具对模型可见。

### 7.7 代码组织调整

第二阶段的新增代码按职责拆分到以下位置：

1. `internal/services/middlewares/toolsearch/tool.go`
   只负责 `tool_search` 元工具本身。
2. `internal/services/middlewares/toolsearch/middleware.go`
   只负责动态工具注入与模型可见性控制。
3. `internal/services/plugin/plugin_search.go`
   只保留语义搜索能力。
4. `internal/services/plugin/plugin_runtime_catalog.go`
   负责运行时工具名映射，不再混在搜索实现里。

同时，agent 侧已删除旧的预搜索残留：

1. `AgentRuntimeConfig.Plugins.SearchQuery`
2. 旧的“读取最后一条用户消息后直接执行插件搜索”的运行时逻辑
3. agent forwarded props 中已无消费方的 `runtimeTypes/minScore/maxMCPTools` 配置解析
