# Avatar Object Access Design

## 1. 背景

当前用户头像上传链路已经能够把文件写入 MinIO，但前端展示失败，根因不是前端渲染，而是后端把 MinIO 访问地址直接作为 `avatar` 返回给前端，并且当前配置中的 `public_base_url` 使用了 `http://127.0.0.1:9000/open-intern`。

这带来两个问题：

- `127.0.0.1` 是后端运行环境自己的回环地址，不是稳定的浏览器外部访问地址。
- 数据库中直接存完整 URL，把“对象标识”和“对象访问策略”耦合在一起，后续切换公开访问、签名 URL、代理下载、CDN 域名时都需要回写历史数据。

因此需要把头像对象的“存储定位”与“访问地址”解耦。

## 2. 目标

- 后端内部继续使用 MinIO 配置中的 `endpoint` 连接对象存储。
- 面向前端的头像字段不再暴露 `127.0.0.1` 这类内部地址。
- 数据库存储稳定对象标识，而不是易变的完整访问 URL。
- 前端继续使用简单的 `avatar` 图片地址渲染，不增加前端复杂度。
- 为后续扩展到签名 URL、代理下载、CDN 域名预留统一边界。

## 3. 非目标

- 不在本轮引入完整的通用媒体中心。
- 不改动无关对象类型的上传语义。
- 不为了兼容旧实现长期保留“双写 URL + key”模式。

## 4. 方案对比

### 方案 A：头像直接改存到 `public/...`

优点：

- 实现最简单，前端可以直接使用稳定公开 URL。
- 缓存与 CDN 接入最直接。

缺点：

- 用户头像永久公开，和 `users/...` 私有对象模型割裂。
- 如果后续希望头像读取具备权限控制或审计能力，需要再次迁移。

### 方案 B：数据库只存对象 key，读接口统一解析为可访问 URL

优点：

- 把“存储标识”和“访问方式”解耦，模型最稳定。
- 前端仍然只拿 `avatar` 字段作为图片地址，接口使用方式不变。
- 后续可在对象存储层统一切换为签名 URL、代理 URL 或 CDN URL。

缺点：

- 需要引入响应 DTO 或序列化转换，避免直接把数据库模型原样返回。
- 需要补一轮现有数据迁移。

### 方案 C：数据库继续存 URL，只把 `public_base_url` 改成外部域名

优点：

- 短期改动最少。

缺点：

- 仍然把数据库绑死到某个具体访问域名。
- 后续更换域名、切 CDN、切签名 URL 时仍然要迁移历史数据。
- 不能解决当前模型边界不清的问题。

## 5. 推荐方案

采用方案 B。

核心原则如下：

- 数据库存储对象 key。
- 对外接口返回可访问 URL。
- URL 解析逻辑只存在于对象存储层。
- MinIO 内部连接地址与浏览器访问地址分离。

## 6. 数据模型设计

### 6.1 用户表字段

本轮不新增字段，继续复用 `user.avatar` 列，但其语义从“头像 URL”调整为“头像对象 key”。

示例：

- 存库值：`users/05b895b0-bb6a-4cfe-ae9a-1f67db169e37/avatar/20260406/63c32df9-a106-487c-b95c-62881861e7db.png`
- 接口返回值：`https://assets.example.com/open-intern/users/...png` 或 `/v1/assets/users/...?expires=...&signature=...`

这样可以避免本轮额外做数据库结构变更，同时把外部接口与内部存储解耦。

### 6.2 代码语义修正

- `models.User.Avatar` 注释改为“头像对象 key”。
- 禁止控制器继续把完整 URL 写入 `avatar` 字段。

## 7. 访问地址模型

### 7.1 内部存储地址

后端连接 MinIO 只使用：

- `minio.endpoint`
- `minio.access_key`
- `minio.secret_key`
- `minio.bucket`

这些配置只属于服务端，不直接返回给前端。

### 7.2 外部访问地址

对象存储层新增统一解析能力，例如：

```go
ResolveObjectAccessURL(objectKey string) (string, error)
```

规则如下：

- `public/...`：返回应用侧稳定资源地址，例如 `/v1/assets/public/...`。
- `users/...`：返回应用侧签名资源地址，例如 `/v1/assets/users/...?...`。

本轮采用“应用侧签名资源 URL”而不是“依赖 Authorization 头的资源代理”，原因如下：

- 当前前端令牌保存在 `localStorage`，浏览器对 `<img src>` 请求不会自动补 `Authorization` 头。
- 应用侧签名 URL 仍然不会暴露 MinIO 内部地址。
- MinIO 读取、签名校验、日志与缓存头控制仍可放在应用层集中处理。

如果后续确认头像天然公开，也可以只改 `ResolveObjectAccessURL` 的实现，把 `users/...` 头像切到公开域名，而不动数据库与前端。

## 8. API 设计

### 8.1 上传头像接口

`POST /v1/users/me/avatar`

调整为：

1. 上传对象到 `users/<user_id>/avatar/...`
2. 将对象 key 写入 `user.avatar`
3. 通过统一解析器生成当前可访问地址
4. 返回：

```json
{
  "key": "users/<user_id>/avatar/20260406/<uuid>.png",
  "url": "/v1/assets/users/<user_id>/avatar/20260406/<uuid>.png?expires=<ts>&signature=<sig>"
}
```

### 8.2 获取当前用户接口

`GET /v1/users/me`

不再直接返回 GORM `models.User` 原始对象，而是返回响应 DTO。DTO 中：

- `avatar` 返回可访问地址
- 不暴露原始对象 key

这样前端接口保持现状，不需要理解对象 key。

### 8.3 登录接口

`POST /v1/users/login`

其中返回的 `user` 也改用相同 DTO，保证登录后写入本地缓存的是可访问头像地址，而不是内部 key 或内部地址。

## 9. 应用代理读取设计

新增资源读取接口，例如：

- `GET /v1/assets/*objectKey`

行为：

1. 校验对象 key 非空且只允许 `users/` 或 `public/` 根前缀。
2. 对 `users/...` 校验 `expires` 与 `signature`。
3. 对 `public/...` 允许直接读取。
4. 从 MinIO 拉取对象流并透传给浏览器。
5. 透传正确的 `Content-Type`。

约束：

- 错误签名或过期签名的 `users/...` 请求必须返回拒绝。
- 不允许访问根前缀之外的任意 key。

## 10. 迁移方案

### 10.1 数据迁移范围

只迁移用户表 `avatar` 字段。

### 10.2 迁移规则

如果字段值匹配当前 MinIO URL 前缀：

- `http://127.0.0.1:9000/open-intern/`
- 或未来配置中的 `public_base_url + "/"`

则将前缀裁剪，只保留对象 key。

示例：

- 迁移前：`http://127.0.0.1:9000/open-intern/users/xxx/avatar/20260406/a.png`
- 迁移后：`users/xxx/avatar/20260406/a.png`

### 10.3 异常数据处理

若 `avatar` 不是空字符串，也不匹配可识别的对象 key / 当前 MinIO URL 前缀，则视为异常历史数据：

- 本轮不做静默兼容解析。
- 迁移脚本输出异常记录，人工确认后处理。

这样可以避免继续掩盖“字段语义混乱”的问题。

## 11. 前端影响

前端页面不需要感知对象 key。

保持：

- 用户页继续直接用 `userInfo.avatar` 渲染头像。
- 工作区布局继续直接用 `userInfo.avatar` 渲染头像。

需要调整的只有一点：

- 默认头像地址来源最终也应与后端返回模型统一，避免默认头像和用户头像分别走两套域名策略。

本轮可以先保持默认头像逻辑不变，因为它本身是 `public/...` 资源；后续再统一到同一个资源地址解析入口。

## 12. 验证方式

完成实现后，至少验证以下内容：

1. 上传头像后，数据库 `user.avatar` 存的是对象 key，不是完整 URL。
2. `GET /v1/users/me` 返回的 `avatar` 是浏览器可访问地址，不包含 `127.0.0.1`。
3. 浏览器直接访问返回的签名头像地址，HTTP 状态为 `200`。
4. 同一路径使用错误签名时，HTTP 状态为 `403`。
5. 默认头像逻辑不因本轮变更退化。

## 13. 风险与取舍

- 应用侧签名资源 URL 会增加应用层带宽消耗，但换来统一的资源出口和不暴露内部地址的访问模型。
- 复用 `user.avatar` 字段可以减少本轮改动，但要求代码层明确其“对象 key”语义，不能再把它当 URL 使用。
- 若后续头像确定允许公开读，可以只替换 URL 解析策略，不需要再次改表或改前端。

## 14. 结论

本轮应把头像访问模型调整为：

- 后端内部：对象 key
- 前端接口：稳定可访问地址
- 存储层：统一负责 key 与 URL 的转换

这样既能解决当前 `127.0.0.1` 与 MinIO `403` 的问题，也能让后续对象存储访问策略具备演进空间。
